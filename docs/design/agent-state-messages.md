# Agent state messages

How non-chat state crosses between the cockpit (and, later, packaged extensions) and the agent ‚Äî in both directions, typed, without polluting the human's chat or the human's stored words.

## Current synthesis

**The defect that opened the thread.** Pi's `input` hook transform is pre-persist: the transformed text _becomes_ the stored user message. The canvas binding used it to prepend `<canvases-open>` / `<canvas-changes>` context, so cockpit context was literally inside the human's persisted entry ‚Äî visible in the chat transcript and breaking the renderer's optimistic-echo confirm-on-text-match ([durable-transcript-identity](../plans/durable-transcript-identity.md) D1). The C1 landed-note's "the human's stored message entry stays verbatim" claim was wrong. Conclusion: the `input` transform is never the channel for cockpit state.

**The channel is custom messages at `before_agent_start`.** Pi's `before_agent_start` handlers can return a message (persisted as its own `CustomMessageEntry`, ordered after the user message) and a chained per-turn `systemPrompt`. With `display: false` the message is hidden from the chat (renderer already gates on `display`) but model-visible. Pi strips `customType` when building LLM context (`convertToLlm` renders customs as plain user-role text), so the kind must live in the text itself: everything flushed in a turn ships as **one `<uix-state>` envelope** (an obvious boundary saying "cockpit state, not human text") containing **one tagged section per type** ‚Äî `<pane-visibility>`, `<canvas-diff>` ‚Äî so each closing tag names what is ending; the whole turn persists as a single `uix.state` entry. Section bodies are freeform per type (JSON for state-shaped payloads, anchored lines for diffs ‚Äî anchored text must stay line-oriented because the model quotes it verbatim into edit-tool args, and JSON-escaping it would invite corruption), and the system-prompt vocabulary section explains the envelope once plus one bullet per tag. This deviates from [persistence-and-session-foundation](../plans/persistence-and-session-foundation.md) C3's "appended before the user message" ordering ‚Äî state messages land after the user message in context ‚Äî accepted for state/diff payloads.

**Change-only via the branch, not a RAM latch.** "Send only when changed" compares the candidate section body against the nearest persisted `uix.state` entry carrying that type's tag up the branch (`getBranch()` available on the handler's `ctx.sessionManager`; entries without the tag mean "unchanged at that turn" and the walk continues past them). The branch itself is the latch, so restart and branch navigation need no re-seeding and the [D3](../plans/durable-transcript-identity.md) rehydrator is not a dependency. Change-only applies to **state-shaped** payloads only; `atTurnBoundary` callbacks are **event-shaped** (and often consuming reads) ‚Äî suppressing an identical payload would lose the event, not delay it ‚Äî so the callback alone decides by returning undefined.

**The substrate primitive this wants to become.** Mirroring how pi builds the system prompt from registered tools (`promptSnippet`/`promptGuidelines`), a state-message registration carries everything one `customType` needs, owned by one module ([one-owner-per-state](../decisions/2026-06-09-one-owner-per-state.md), composition-root rule):

- `registerStateMessage({ customType, schema, description, policy })` ‚Äî TypeBox schema validated at the boundary (a failure is an app bug and throws; the agent never sees garbage), `description` is the vocabulary partial, `policy` covers change-only vs always.
- `emitState(customType, payload)` ‚Äî **push fire-and-latch** as the default: extensions emit at the moment state changes; the substrate latches latest-per-type and flushes at the turn boundary. A registration may instead supply a **boundary callback** for compute-at-submit state ‚Äî the canvas human-diff is the forcing case, because the channel's `consumeChanges()` is a consuming read that must run exactly once at the boundary.
- **One substrate-owned `before_agent_start` assembler** builds the single vocabulary section (registration order = composition-root order) and flushes latched state ‚Äî not per-binding `systemPrompt` chaining.
- **Namespacing enforced by the handle, not convention:** substrate keys are `uix.*`; an extension's API handle prefixes its customTypes from the package name (the `DisposableBag` pattern), so extensions structurally cannot collide or impersonate.
- Later, the same registration is the natural home for the D3 rehydration reducer per key.

**Landed 2026-06-11** as `createStateMessages()` in `src/main/agent/state-messages.ts` ‚Äî `register`/`emit` plus an assembler `AgentBinding` ordered last in the composition root ‚Äî with the canvas's two messages as first consumer. Known constraint accepted at landing: registration is composition-time only (`register` throws once the assembler installs); extension hot-reload will need late registration, a contained change (capture the live `pi` handle, install the flush handler at `register()` time).

**Agent ‚Üí app structured output is a tool, not a third mechanism.** A registered tool whose parameter schema _is_ the output contract gives schema enforcement and retry natively: structural mismatches are rejected at the tool boundary, semantic failures return an `isError` result with a precise message, and the model retrying a failed call is the existing agent loop. The tool's execute **commits the validated payload to the content store and the pane hydrates from the store** ‚Äî agent writes an artifact, UI renders it, preserving [no-agent-ui-manipulation](../decisions/2026-05-30-no-agent-ui-manipulation.md). This is the canvas pattern with a schema'd state doc substituted for HTML ‚Äî the bridge to a case-2 (lit/signals-style) canvas that passes state instead of raw HTML diffs ([canvas-data-channel](./canvas-data-channel.md)). If a future case truly cannot be a tool (output as the model's final text), design that against the concrete consumer.

**Extension packaging is the direction of travel.** The canvas (tools + pane + state messages + vocabulary) becomes a comes-with, replaceable extension contributing via the `pi` and `uix` fields; the conversation pane likewise (existing backlog seed). The pane host and shell stay substrate ‚Äî like pi's TUI, which is host library, not extension. Shaping `registerStateMessage`/`emitState` so they move to `@uix/api` verbatim is the dogfooding test that the API is sufficient. **The agent session is substrate; the conversation is one view of it** ‚Äî a shipped app may have no chat pane at all (dropdowns assemble prompts behind the scenes), which makes state messages the _input primitive_ for non-chat apps, not a canvas nicety. The conversation pane doubles as the dev-mode **inspector**: a menu toggle rendering `display: false` customs (and other hidden entry streams) is a renderer-only change on the existing transcript items.

**CustomEntry as an event log.** `pi.appendEntry` is the model-invisible sibling (ignored by `buildSessionContext`, skipped by transcript projection, branch-aware) ‚Äî usable as an append-only user-action log on the same substrate, folded by D3 reducers (accumulate-all for logs, last-wins for state), with no LLM turn triggered. Caveats: keep it low-frequency (pi loads the whole session file into memory ‚Äî log submitted actions, not keystrokes); `appendEntry` parents to the current leaf with no `parentId` parameter, so concurrent forks or inspect-one-branch-while-acting-on-another need leaf discipline pi doesn't yet provide; and a reducer that becomes load-bearing state needs a single writing owner like any `uix.*` key.

**Fan-out / one-off mode (later).** Non-conversational invocations are "session with one turn": a **prepared root** per app variant (system prompt + tool contracts + vocabulary + seeded state entries) and a fork per invocation that runs once and never rejoins ‚Äî possibly multiple registered roots, one session file per root as the natural retention unit. Every run is recorded and inspectable as a sibling branch; identical prefixes align with provider prompt caching; change-only state compares against the root's seeded entries unchanged. What needs design then is lifecycle (fork-per-invocation creation/disposal, concurrency), not messaging.

## Open questions

- Per-type vocabulary partials vs one substrate-templated section rendered from registrations (schema ‚Üí shape line), once there are enough types to compare.
- Late registration for extension hot-reload (today `register` is composition-time only; see synthesis).
- The rich human-facing rendering of `uix.canvas-diff` (the "you changed these lines" chat strip) ‚Äî the message ships `details` structured hunks for it, but the block renderer is unbuilt.
- Leaf discipline for concurrent appends (fan-out mode, multi-pane action logs) ‚Äî needs pi cooperation (`parentId` on append) or a manager-per-fork pattern.

### 2026-06-12 ‚Äî substrate refactor design: handle-based registration, driver-owned install, bag lifetimes

Designed (not yet built) the next shape of `createStateMessages`, fixing the anti-pattern that the landed version mixes two audiences in one type and that its lifetimes don't line up with the bag model. **This entry records the converged design; the synthesis and "Landed 2026-06-11" note above describe the _old_ shape and are stale pending this build** (they still say `register`/`emit` + an assembler `binding` ordered last + composition-time-only registration; the build replaces all three and resolves the "late registration" open question).

**The anti-pattern.** `StateMessages` serves two audiences in one object: `register`/`emit` are registrant-facing (and head for `@uix/api`), while `binding` is composition-root wiring an extension must never touch ‚Äî nothing stops a registrant calling `stateMessages.binding(pi)` and double-installing. The deeper smell is _lifetime_: the store (registrations) is app-scoped, but the installed `before_agent_start` handler + its vocabulary string are session-scoped, and they're collapsed into one app-lifetime closure ‚Äî which is _why_ `register` has to throw forever and reload can't change the registration set.

**`register` returns a handle; `emit` lives on it.** `register(config)` returns a Disposable **handle**. The capability is the object: only the owner can emit its own type (no stringly `emit("uix.pane-visibility", ‚Ä¶)` on a shared store), the schema types the emit payload, and disposing the handle removes the registration _and_ its latch. The handle is dropped in the **registrant's own bag** ‚Äî the per-extension `DisposableBag` the loader already builds (`extensions/loader.ts`), or the driver-scope bag for the core canvas binding. This is the lifetime-bags substrate primitive doing its job; extension reload disposes the bag ‚Üí registrations vanish.

**One delivery mechanism, push XOR pull.** `emit` and the old `atTurnBoundary` collapse: a registration either **receives** a function (`emit`, push ‚Äî substrate latches latest-per-type, semantics are _replace_, only safe for state) or **provides** one (`read`, pull ‚Äî substrate calls it at the boundary, the return enters the same pipeline, the only home for consuming/event-shaped reads like the canvas diff). The latch is just the degenerate read ("return what was last pushed"); internally `value = reg.read ? await reg.read() : latch`. **`policy` simplifies to pure suppression**: `change-only` vs `every-turn`, with **safe defaults** ‚Äî push (latch) defaults `change-only`, a `read` callback defaults `every-turn`, so a consuming read is never suppressed unless its owner explicitly opts in (the event-loss regression the landed code guards with `!reg.atTurnBoundary` becomes declarative). Trigger-ownership and state-vs-event turn out to be the same axis: push‚áíreplace‚áístate, pull‚áíconsume‚áíevents.

**The driver owns the install seam; no `binding` on the public type.** Registrants never need the binding ‚Äî it existed only because the `agentBindings` list was the _only_ pipe to the live pi handle, an artifact of plumbing. Resolution: `createAgentDriver({ ‚Ä¶, stateMessages })` installs the flush handler itself when it builds the pi session. `index.ts` shrinks to `const stateMessages = createStateMessages()` passed both to the canvas binding (narrow `register` face) and to the driver (install seam); the "assembler ordered last" comment and its test both die, because registration is decoupled from install. **Install snapshots** the current registrations and computes the vocabulary string **once** (byte-stable per session; `cache_control` rides the last user message, so this is structural cache-safety, not incidental). Late registration becomes legal ‚Äî it just takes effect at the next install ‚Äî so **hot-reload falls out for free**: extension bag disposes ‚Üí re-activation re-registers ‚Üí `driver.reload()` reinstalls ‚Üí new snapshot at a user-caused boundary. (Verify when building: that pi's `session.reload()` actually re-runs the extension factory / re-fires the install; if not, the re-snapshot needs a different hook.)

**Shared early-stopping branch walk.** Today `lastPersistedBody` walks leaf‚Üíroot once _per_ change-only registrant. The refactor does **one** walk that collects the nearest persisted body for every change-only registrant and stops as soon as all are resolved (or hits root) ‚Äî only possible because diffing is substrate-owned.

**Where extensions get pi (the principle behind all of this).** Raw pi reaches extensions in **exactly one place**, never sprinkled across wrapped concepts: an extension wanting raw agent capability declares a `pi` field and **pi loads it natively** (the driver's resource loader already discovers user pi resources alongside the core extension). `@uix/api` carries only **cockpit-mediated** concepts ‚Äî things whose value is in the cockpit's assembly, like state messages (`register`/`emit`), later block actions. The `agentBindings` list stays **cockpit-core-only** ("how the cockpit talks to the agent at all", AGENTS.md), never an extension point.

**Cross-thread build order** (so it survives the session): **#1 this refactor** (ready, no new pi deps) ‚Üí **#2 docs** (this entry + the delivery-mechanics entry below) ‚Üí **#3 canvas `base` frontier + conflict guard + read-advances-frontier + anchor-range reads** ([canvas-data-channel](./canvas-data-channel.md) 2026-06-12; gated only on introducing `base` as tracked state) ‚Üí **#4 steer tier** (peek/commit-on-persist; needs #1 + #3 + the identity persistence callback, which exists) ‚Üí **#5 substrate followUp / app-continuation queue** (needs #1 + `agent_end` handling). Independent track, not displaced: durable-transcript-identity **D2/D3**, the **pane host** (gates canvas-as-extension), and the **`@uix/api` extraction** (gated on the pane host ‚Äî _not_ imminent).

### 2026-06-12 ‚Äî verified pi delivery mechanics, and the delivery doctrine they justify

Designing immediate (mid-run) state delivery forced reading pi's actual loop from dist (`@earendil-works/pi-agent-core/dist/agent-loop.js`, `pi-coding-agent/dist/core/agent-session.js`, `pi-ai/dist/providers/anthropic.js`). Recording the mechanics because they were expensive to derive, non-obvious, and load-bearing for every delivery unit ahead.

**Verified pi facts:**

- **The LLM is stateless; the runtime owns the message array.** Each call sends the whole array and returns one assistant message. A tool call is the assistant message _ending_ with `tool_use` blocks ‚Äî generation stops, the runtime executes the tools, appends results, and makes a fresh call. "Continuing after a tool" is conditioning on the longer array, not a resumed stream; the prior assistant message is immutable history and is never reopened.
- **Tool results are user-role messages, and so is everything the runtime injects.** The Anthropic serializer renders `toolResult` as a user message of `tool_result` blocks (correlated by `tool_use_id`, not position); consecutive tool results coalesce into one user message, and an injected/steered message is a _separate_ user message after them (the API allows consecutive user messages). "User role" means "not the model's output," not "the human typed it" ‚Äî the identical channel our `uix.state` envelope rides via `convertToLlm`, which is exactly _why_ the kind must live in the content text (the XML tag), since the role can't carry it.
- **Run vs turn.** A **turn** = one LLM call + executing its tools; a **run** = the loop until the assistant produces no tool calls and no queued messages remain (`agent_start`‚Üí`agent_end`). The model only ever experiences turns; the run exists only in the runtime's while-loop.
- **Four delivery seams, and what each does to state.** `before_agent_start` fires **once per run** inside `prompt()`, beside the user message ‚Äî our flush rides it. **steer** (`sendCustomMessage`/`prompt` with `deliverAs: "steer"`) drains at the top of the next inner-loop iteration ‚Äî _between_ turns, after the current tool batch, before the next LLM call ‚Äî and **bypasses `before_agent_start`**. **followUp** drains only when the run would otherwise stop, _extends the run_ with more turns, and also **bypasses `before_agent_start`**. **nextTurn** parks in `_pendingNextTurnMessages` and is injected beside the _next_ `prompt()`'s user message (next run). Consequence: only a run-boundary flush ‚Äî or a steer we pair by hand ‚Äî carries _fresh_ state; followUp and nextTurn structurally cannot attach a contemporaneous flush.
- **`cache_control` sits on the last user message.** The conversation prefix is cached, so byte-stable prefixes matter ‚Äî the concrete reason to snapshot the system-prompt vocabulary **once at install** rather than recompute it per turn (recompute is byte-stable only incidentally; snapshot makes it structural).

**The delivery doctrine these justify** (decision, for when the delivery units build):

- **steer** ‚Äî pi's queue, but always shipped as a _pair_: the state flush (`deliverAs: "steer"`) immediately followed by the human message, never the message alone. This is the only way mid-run human input arrives with the canvas state it refers to.
- **followUp** ‚Äî a **substrate-owned** queue, delivered via a fresh `prompt()` at `agent_end`, so it gets the full run-boundary treatment (fresh diff, change-only, vocabulary). Pi's native followUp goes **unused by UIX**: it bypasses `before_agent_start`, so a queued human message would reach the model against stale state ‚Äî disconnected from the canvas edits made beside it.
- **nextTurn** ‚Äî **never**; strictly dominated by computing the flush at `before_agent_start` (same reading position, but fresh and change-only checked instead of parked-and-possibly-stale).
- **app continuation** ‚Äî the same substrate-owned queue as followUp; "run finished ‚Üí substrate decides the next invocation" is a cockpit (pilot) decision and composes with the fan-out direction, where a pi-side queue wouldn't.
- **The invariant underneath all of it:** _a human message never reaches the model without contemporaneous cockpit state._ Every moment a human utterance enters the model's reading order, the current canvas truth must enter beside it ‚Äî which is what disqualifies both pi-native followUp and nextTurn for UIX.

The steer tier's peek/commit-on-persist protocol and the `base`-as-observation-frontier primitive it relies on are specified on the channel side ([canvas-data-channel](./canvas-data-channel.md), 2026-06-12).

### 2026-06-11 ‚Äî framing, from the `<canvases-open>`-in-chat defect

Thread opened after tracing why cockpit context showed inside the user's chat message: pi's `input` transform rewrites the persisted user entry (dist source: `prompt()` builds the user message from the transformed text), falsifying C1's verbatim claim. Settled in discussion: `display: false` custom messages at `before_agent_start` as the channel, with a debug toggle (not default display) for inspection; system-prompt vocabulary so payloads stay compact ‚Äî required, since `convertToLlm` strips `customType`; change-only computed against the branch (the branch is the latch); push `emitState` fire-and-latch as the default mechanism with a boundary-callback option for consuming reads (the canvas diff); agent‚Üíapp structured output as tools-as-contracts with store-commit hydration, explicitly rejecting a third "schema-validated custom message reply" mechanism; canvas and chat as comes-with extensions over a `registerStateMessage` substrate primitive, customTypes namespaced by the API handle; CustomEntry streams as model-invisible event logs; fan-out prepared-root mode as the eventual non-chat shape. First slice considered hand-assembling in the canvas binding and extracting the primitive at the second consumer, but the primitive's plumbing (vocabulary assembly, branch comparison, display gating) is registrant-agnostic from day one, so `createStateMessages` landed immediately with the canvas's `uix.pane-visibility` (change-only, via `emit`) and `uix.canvas-diff` (consuming read, via `atTurnBoundary`) as first consumers. The vocabulary section is emitted only when at least one registration exists.
