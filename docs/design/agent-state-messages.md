# Agent state messages

How non-chat state crosses between the cockpit (and, later, packaged extensions) and the agent — in both directions, typed, without polluting the human's chat or the human's stored words.

## Current synthesis

**The defect that opened the thread.** Pi's `input` hook transform is pre-persist: the transformed text _becomes_ the stored user message. The canvas binding used it to prepend `<canvases-open>` / `<canvas-changes>` context, so cockpit context was literally inside the human's persisted entry — visible in the chat transcript and breaking the renderer's optimistic-echo confirm-on-text-match ([durable-transcript-identity](../plans/durable-transcript-identity.md) D1). The C1 landed-note's "the human's stored message entry stays verbatim" claim was wrong. Conclusion: the `input` transform is never the channel for cockpit state.

**The channel is custom messages at `before_agent_start`.** Pi's `before_agent_start` handlers can return a message (persisted as its own `CustomMessageEntry`, ordered after the user message) and a chained per-turn `systemPrompt`. With `display: false` the message is hidden from the chat (renderer already gates on `display`) but model-visible. Pi strips `customType` when building LLM context (`convertToLlm` renders customs as plain user-role text), so **payloads must be self-identifying and the system prompt carries the vocabulary**: a registered section explains each message shape once, and per-turn payloads stay compact (e.g. `{"canvases_open": ["main"]}`). This deviates from [persistence-and-session-foundation](../plans/persistence-and-session-foundation.md) C3's "appended before the user message" ordering — state messages land after the user message in context — accepted for state/diff payloads.

**Change-only via the branch, not a RAM latch.** "Send only when changed" compares the candidate payload against the nearest persisted entry of the same customType up the branch (`getBranch()` available on the handler's `ctx.sessionManager`). The branch itself is the latch, so restart and branch navigation need no re-seeding and the [D3](../plans/durable-transcript-identity.md) rehydrator is not a dependency.

**The substrate primitive this wants to become.** Mirroring how pi builds the system prompt from registered tools (`promptSnippet`/`promptGuidelines`), a state-message registration carries everything one `customType` needs, owned by one module ([one-owner-per-state](../decisions/2026-06-09-one-owner-per-state.md), composition-root rule):

- `registerStateMessage({ customType, schema, description, policy })` — TypeBox schema validated at the boundary (a failure is an app bug and throws; the agent never sees garbage), `description` is the vocabulary partial, `policy` covers change-only vs always.
- `emitState(customType, payload)` — **push fire-and-latch** as the default: extensions emit at the moment state changes; the substrate latches latest-per-type and flushes at the turn boundary. A registration may instead supply a **boundary callback** for compute-at-submit state — the canvas human-diff is the forcing case, because `collectChanges()` is a consuming read that must run exactly once at the boundary.
- **One substrate-owned `before_agent_start` assembler** builds the single vocabulary section (registration order = composition-root order) and flushes latched state — not per-binding `systemPrompt` chaining.
- **Namespacing enforced by the handle, not convention:** substrate keys are `uix.*`; an extension's API handle prefixes its customTypes from the package name (the `DisposableBag` pattern), so extensions structurally cannot collide or impersonate.
- Later, the same registration is the natural home for the D3 rehydration reducer per key.

**Landed 2026-06-11** as `createStateMessages()` in `src/main/agent/state-messages.ts` — `register`/`emit` plus an assembler `AgentBinding` ordered last in the composition root — with the canvas's two messages as first consumer. Known constraint accepted at landing: registration is composition-time only (`register` throws once the assembler installs); extension hot-reload will need late registration, a contained change (capture the live `pi` handle, install the flush handler at `register()` time).

**Agent → app structured output is a tool, not a third mechanism.** A registered tool whose parameter schema _is_ the output contract gives schema enforcement and retry natively: structural mismatches are rejected at the tool boundary, semantic failures return an `isError` result with a precise message, and the model retrying a failed call is the existing agent loop. The tool's execute **commits the validated payload to the content store and the pane hydrates from the store** — agent writes an artifact, UI renders it, preserving [no-agent-ui-manipulation](../decisions/2026-05-30-no-agent-ui-manipulation.md). This is the canvas pattern with a schema'd state doc substituted for HTML — the bridge to a case-2 (lit/signals-style) canvas that passes state instead of raw HTML diffs ([canvas-data-channel](./canvas-data-channel.md)). If a future case truly cannot be a tool (output as the model's final text), design that against the concrete consumer.

**Extension packaging is the direction of travel.** The canvas (tools + pane + state messages + vocabulary) becomes a comes-with, replaceable extension contributing via the `pi` and `uix` fields; the conversation pane likewise (existing backlog seed). The pane host and shell stay substrate — like pi's TUI, which is host library, not extension. Shaping `registerStateMessage`/`emitState` so they move to `@uix/api` verbatim is the dogfooding test that the API is sufficient. **The agent session is substrate; the conversation is one view of it** — a shipped app may have no chat pane at all (dropdowns assemble prompts behind the scenes), which makes state messages the _input primitive_ for non-chat apps, not a canvas nicety. The conversation pane doubles as the dev-mode **inspector**: a menu toggle rendering `display: false` customs (and other hidden entry streams) is a renderer-only change on the existing transcript items.

**CustomEntry as an event log.** `pi.appendEntry` is the model-invisible sibling (ignored by `buildSessionContext`, skipped by transcript projection, branch-aware) — usable as an append-only user-action log on the same substrate, folded by D3 reducers (accumulate-all for logs, last-wins for state), with no LLM turn triggered. Caveats: keep it low-frequency (pi loads the whole session file into memory — log submitted actions, not keystrokes); `appendEntry` parents to the current leaf with no `parentId` parameter, so concurrent forks or inspect-one-branch-while-acting-on-another need leaf discipline pi doesn't yet provide; and a reducer that becomes load-bearing state needs a single writing owner like any `uix.*` key.

**Fan-out / one-off mode (later).** Non-conversational invocations are "session with one turn": a **prepared root** per app variant (system prompt + tool contracts + vocabulary + seeded state entries) and a fork per invocation that runs once and never rejoins — possibly multiple registered roots, one session file per root as the natural retention unit. Every run is recorded and inspectable as a sibling branch; identical prefixes align with provider prompt caching; change-only state compares against the root's seeded entries unchanged. What needs design then is lifecycle (fork-per-invocation creation/disposal, concurrency), not messaging.

## Open questions

- Per-type vocabulary partials vs one substrate-templated section rendered from registrations (schema → shape line), once there are enough types to compare.
- Late registration for extension hot-reload (today `register` is composition-time only; see synthesis).
- The rich human-facing rendering of `uix.canvas-diff` (the "you changed these lines" chat strip) — the message ships `details` structured hunks for it, but the block renderer is unbuilt.
- Leaf discipline for concurrent appends (fan-out mode, multi-pane action logs) — needs pi cooperation (`parentId` on append) or a manager-per-fork pattern.

## Log

### 2026-06-11 — framing, from the `<canvases-open>`-in-chat defect

Thread opened after tracing why cockpit context showed inside the user's chat message: pi's `input` transform rewrites the persisted user entry (dist source: `prompt()` builds the user message from the transformed text), falsifying C1's verbatim claim. Settled in discussion: `display: false` custom messages at `before_agent_start` as the channel, with a debug toggle (not default display) for inspection; system-prompt vocabulary so payloads stay compact — required, since `convertToLlm` strips `customType`; change-only computed against the branch (the branch is the latch); push `emitState` fire-and-latch as the default mechanism with a boundary-callback option for consuming reads (the canvas diff); agent→app structured output as tools-as-contracts with store-commit hydration, explicitly rejecting a third "schema-validated custom message reply" mechanism; canvas and chat as comes-with extensions over a `registerStateMessage` substrate primitive, customTypes namespaced by the API handle; CustomEntry streams as model-invisible event logs; fan-out prepared-root mode as the eventual non-chat shape. First slice considered hand-assembling in the canvas binding and extracting the primitive at the second consumer, but the primitive's plumbing (vocabulary assembly, branch comparison, display gating) is registrant-agnostic from day one, so `createStateMessages` landed immediately with the canvas's `uix.pane-visibility` (change-only, via `emit`) and `uix.canvas-diff` (consuming read, via `atTurnBoundary`) as first consumers. The vocabulary section is emitted only when at least one registration exists.
