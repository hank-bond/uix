# Canvas data channel

## Current synthesis

**Frame.** A pane is a view/editor over a **document in a content store**, not over a filesystem path. The canvas is the first case-1 document surface: the agent pushes changes in, the human pushes changes back, and UIX's job is **efficiency + reliability** at that crossing. The shared language _for the canvas feature's agent tools_ is anchored text: agent→document is assigned-anchor edits; document→agent is assigned-anchor diffs of what changed outside the agent's observed write path. Anchors are not canvas semantics and not a universal document substrate; they are a reusable, mechanical agent-editing projection that the canvas feature contributes because it is the right way for the agent to work with canvas HTML.

**Canvas is a default feature, not privileged core.** UIX ships a useful default app configuration with chat and canvas enabled, but those are first-party features consuming substrate contribution points, not hardcoded laws of the cockpit. A real app can disable or replace canvas and chat. That means canvas-owned identifiers use the feature namespace (`canvas.*`), while `uix.*` stays reserved for substrate lifecycles and built-in protocol/state types.

**Document kind, not owner.** A document has an intended semantic purpose — its **kind** — that defines the canonical write contract for that document. The canvas feature registers a canvas-HTML kind for canvas document ids: every write candidate, no matter whether it came from the pane, an anchored agent tool, a future XPath tool, or restore, is normalized/validated by that kind before it becomes stored truth. There is no public normalization pipeline for now: one kind decides the final bytes for its document purpose. Other contributors can listen to changes or provide edit lenses, but they do not mutate the canonical result after the kind.

**Exact source ids drive side effects.** Document writes carry an exact `sourceId`, not a broad `human`/`agent` category. Feature-originated writes use feature namespaces such as `canvas.pane.writeback`, `canvas.agent.anchor_write`, and `canvas.agent.anchor_edit`; substrate lifecycles use `uix.*` sources such as `uix.document.restore` or `uix.document.preview`. Listeners key off those exact sources: canvas pane writeback should not refresh the pane but should make an agent-visible diff available; agent anchor edits should refresh the pane but should not queue a human/canvas diff back to the agent.

**Two cases of canvas content** (boundary = is structure a per-turn variable or a frozen dependency?):

- **Case 1 — document / notebook.** Structure and data are fused; the agent edits the fused thing every turn (e.g. a PR rendered as summary + diff + inline discussion). Raw HTML, doc-as-state, free hydration. _This is all we build now._
- **Case 2 — application.** Structure frozen into a versioned frontend extension; the agent mostly edits a _state document_. Reusable; needs pi + frontend extensions. The wiki tool is a case-2 hypothesis.
- **Promotion path:** vibe a case-1 notebook repeatedly → the app shape emerges → graduate to case 2. Grown from observed patterns, not designed top-down.
- **Litmus that keeps them apart:** case 1 never gets a shipped runtime or a state store separate from its hydrated document. Want either → you've graduated. (This is why React/state-store panes are wrong for case 1 but right for case 2 — same free-hydration tradeoff read from both ends.)

**Document store, not filesystem.** A pane is a view/editor over a document addressed by resource id (`doc://canvas/main`, `doc://report/foo`) behind a **document-store interface** — `getCurrent(resourceId)` / `setCurrent(resourceId, content)` / `createSnapshot(...)` / `getVersion(...)` — owned by the cockpit, never a filesystem path. Workspace files use their own cwd-relative resource scheme later (`workspace://src/main.ts`); today canvas documents are managed `doc://` resources stored under `.uix`. The channel only needs _current content + diff_; whether the store keeps history is hidden behind this seam (which is also the **hosting boundary** — a local file/object store now, DB/object-store + pub/sub when hosted; see [hosting-compatible-by-default](../decisions/2026-05-31-hosting-compatible-by-default.md)). The first cut is a **trivial single-version store**; the git-backed versioned store slots in later without touching the channel (see [pane-and-file-versioning](./pane-and-file-versioning.md)). Content hashes are good object ids but are **not** line anchors. The agent never addresses store objects directly; to transform pane content with `bash` it pulls via `canvas__anchor_read`, writes scratch/temp files, then updates current content via `canvas__anchor_write`/`canvas__anchor_edit`.

**Editor state vs versioning are different layers.** The anchored editor's state — the anchor↔line map plus the last-observed text it diffs against — is session-scoped and _regenerable from current content_. It is **not** history and is not persisted as authority — its durable home is the **version commit meta** in the document store (a version is `{contentRef, meta: {anchorMap, allocIndex}}`, so checkout restores content and anchors as one atomic, necessarily-consistent unit; meta absent → regenerate, renumbered, match-guard as last resort). See Log 2026-06-09, which supersedes the droppable-sidecar cache from Log 2026-06-04. Versioning (append-only commits, conversation-node pointers, rollback) lives **behind the document-store interface**, so it can land later without touching the editor: rollback restores _content_, and the editor re-derives anchors. The whole history/rollback design — git as backend, hash parentage, the two-repo split, per-run user-file snapshots, and the conversation-node meta contract — is its own thread: [pane-and-file-versioning](./pane-and-file-versioning.md).

**The anchored edit model** (adopt Dirac's shape wholesale — see [the Dirac post](https://dirac.run/posts/hash-anchors-myers-diff-single-token)):

- **Assigned compact anchors**, not content hashes. Content hashes are long, collide on duplicate lines when truncated, and are ephemeral; assigned IDs are drawn from a curated anchor pool, are collision-free, and stay stable across content edits. The pool is model-agnostic; token efficiency is nice-to-have, not a support gate. The tradeoff (a stateful, session-scoped anchor→line map) is acceptable because the map is regenerable from the doc, not persisted — so it doesn't make the filesystem load-bearing.
- **A Myers reconciler** reassigns anchors only to changed lines after each edit, and runs identically for human pane edits and filesystem external changes. The state manager keeps the last observed text as the diff base; if `bash`/editor/git mutates a normal file, the next anchored `read`/`edit` reconciles against disk and can report changed hunks with fresh anchors.
- **Edit op** = `{ start_line, end_line, replacement }` with an inclusive line range (one line for a single boundary; two boundaries span a range); the model emits only replacement content. Each boundary is the **full rendered line** (`<anchor>§<text>`), not a bare anchor — the text half is a verbatim guard (validate by string-match: reject unless the live line behind the anchor still matches). **Every edit/read result returns fresh anchors for touched lines**, so the agent never re-reads to learn current anchors. There are no zero-width ranges and no `insert_before`/`insert_after`; insertions are expressed by replacing an adjacent inclusive range with replacement content that includes retained line(s) plus inserted line(s).
- **Operations split by intent — `write` clobbers, `reconcile` diffs.** `read` returns the current snapshot (lines); `edit` diffs only its target range so unchanged in-range lines keep their anchors. The two whole-document operations are deliberately different: `reconcile(text)` runs Myers against the whole doc to **preserve unchanged anchors and emit the human-diff** — it is the writeback / external-change path (and the only one that returns change hunks for a whole-doc replacement). The agent's pi `write` tool is a **clobber**: it replaces the document wholesale, so it rebuilds from scratch with **no diff** (cost O(new lines), not O(N·D)) and returns fresh lines, because diffing a wholesale rewrite would only ever report "everything changed." Returning either lines _or_ change hunks (never both) keeps each op's result single-purpose.
- **Anchors are unique for the life of a session.** A clobber does **not** reset allocation to the start of the pool — it continues from the running index, so the rewritten version shares no anchor with the version it replaced. This matters because both can be live in one chat at once (e.g. the agent's earlier read and the post-`write` document), and a reused anchor naming two different lines across versions would be ambiguous. Monotonic allocation is safe: the pool's `size + size²` capacity (millions of anchors) dwarfs what any single conversation could reference within a context window.

**Tool surface.** Anchored editing is one agent-facing lens over documents, not the only possible document tool. The canvas feature currently contributes canvas-named tools — `canvas__anchor_read`/`write`/`edit` over a canvas key — and those tools activate/maintain the anchor projection for the touched canvas document. Future XPath, JSONPath, AST, or format-specific tools can edit the same document through the same document engine; because all writes pass through the document kind, their final result is normalized the same way and active projections can update from the committed canonical content. Normal filesystem tools may keep pi's default search/replace semantics unless a feature opts particular files into managed document semantics or contributes replacement tools.

For agent writes, normalization is folded into the tool result. The candidate the model supplied is not treated as durable truth; the document kind normalizes it, the canonical snapshot is committed, the anchor projection updates to that final content, and the tool result returns the final anchored canonical diff/lines. Later context trimming can drop the raw write input because the result is the authoritative model-visible state. Normalization is semantically neutral, so exposing the canonical result as "what you wrote" is a steering advantage: subsequent model edits tend to move closer to the document kind's canonical format. If we need to debug contributed normalization code, the normalization diff can live in snapshot metadata or developer UI; it does not belong in the generic write event or ordinary agent context.

**Human pane changes.** Human edits go through the same document engine write path and arrive over **internal eventing, not a filesystem watcher** — so the agent's own writes never echo back as phantom human changes. The current iframe shim listens for native `input` / `change` events, reflects form-control state into serialized HTML, and also captures author-provided `contenteditable` regions; it does **not** set global `contenteditable` or make arbitrary canvas elements selectable/editable/deletable as an editing surface. Case-1 panes are fully hydrated documents: clicking a UI control that adds an element changes the document itself. A pane writeback uses source `canvas.pane.writeback`: the document kind normalizes the HTML and stores the canonical snapshot/latest; the canvas pane does not need to refresh just because its own write normalized, because the normalization is semantically neutral and the next load reads canonical truth. If an anchor projection is active for the document, it reconciles against the canonical post-write content and can later produce anchored hunks.

The mutable latest canvas is the current working HTML document; immutable snapshots are created at run boundaries and become durable/traversable only when a `uix.turn-state` `CustomEntry` points at them. At user submit, before the user message is appended, UIX creates a snapshot from latest, diffs the nearest upstream turn-state snapshot → the new normalized snapshot, appends the UIX-private turn-state pointer, then appends the model-visible hidden `uix.state`/`<canvas.canvas-diff>` `CustomMessageEntry` containing anchored hunks when canvas writeback produced changes the agent has not already observed. Agent writes update latest plus active projections during the run; at `agent_end`, if canvases changed, UIX creates a snapshot from the final latest canvas and appends a post-run `uix.turn-state` so the next user diff starts from what the agent already saw. Opportunistic developer preview diffs may run on `canvas.pane.writeback` and render in the agent pane, but they are UI/debug-only, may be anchorless, and clear on submit; the agent-bound diff is the submit snapshot transaction.

**Anchor pool = committed static asset** (not in the harness; the harness loads a committed pool). The first landed pool is vendored from Dirac's Apache-2.0 `src/utils/.hash_anchors` list and attributed in `src/main/anchors/assets/README.md`. The runtime pool is model-agnostic: UIX uses the same anchors for every model and does not gate by tokenizer support; because the pool is small, runtime keeps it in memory, and each managed document stores an allocation index. Allocation uses single words first, then composes two-word anchors with naive row-major pair indexes after the single-word pool is exhausted; the pool asset is pre-sorted so early two-word anchors stay compact. Provider-specific probing can be added later as an optimization, but no provider is locked out.

**Context efficiency, two halves:**

- _Half A_ — return the canonical, anchored doc/excerpt in the **tool result**. Ships now via `customTools`; gives anchors with no read-back.
- _Half B_ — drop the redundant raw write-input. Feasible via pi's `context` hook ("Fired before each LLM call. Can modify messages"), which is non-destructive (transforms what's _sent_, leaves pi's session honest). Put truth in the result, never falsify the assistant turn → no disclosure needed. Truncate (don't substitute) the raw input on later turns.
- **Caching discipline:** the doc goes through the model once → becomes a stable cached prefix → edits are append-only deltas; transforms must be deterministic to keep the prefix byte-stable; re-materialize only at compaction.

**Integration.** Use the **lower-level pi path** (`createAgentSessionFromServices` / runtime) behind **one thin adapter** (a single blast-radius for pi-version drift). An **agent facet** is UIX-core's internal Pi-facing boundary (tools + context transforms + prompt sections + hooks + lifetime); shared state (reconciler, document store, filesystem adapter, canvas store) is injected as **typed services**, not facet-to-facet reach-through. Two tiers stay distinct: substrate facets (deep, trusted) vs user extensions (narrow, jiti-loaded). This is the "kernel module vs userland" split.

**Thesis.** UIX is a context-engineering layer that owns the agent loop and curates the tool-call/result history for bidirectional document side-channels — and owning the loop is what lets it be more efficient and reliable than a generic FS agent or MCP tool.

### 2026-06-21 — canvas as feature; document kinds, source ids, and projections

Settled the featurified canvas model. Canvas is a bundled default feature, not privileged core, and it contributes a document kind plus its pane, tools, listeners, and state behavior. The kind is the semantic write contract: canvas HTML is canonicalized on every write regardless of whether the writer is pane writeback, an agent anchor tool, a future XPath tool, or restore. Anchors are a mechanical projection activated by agent tool usage, not a canvas concept; if active, they update from any committed canonical write and ride snapshot metadata for continuity.

Document writes should emit small ref-based events, not payload copies: `{ resourceId, kindId, sourceId, beforeSnapshotId, afterSnapshotId, normalized }`. Source ids are exact and namespaced by contributor (`canvas.pane.writeback`, `canvas.agent.anchor_edit`, `canvas.agent.anchor_write`) while substrate lifecycles keep `uix.*` (`uix.document.restore`, `uix.document.preview`). Listeners decide side effects from source: pane writeback queues/derives agent-visible canvas diffs and does not refresh the pane; agent tool writes refresh the pane and fold normalization into the tool result rather than sending a separate "normalizer changed it" message.

### 2026-06-21 — `DocumentChannel` → `CanvasDocumentBuffer`; buffer/store/anchoring layering

Renamed `DocumentChannel` through `DocumentBuffer` to `CanvasDocumentBuffer` (`src/features/canvas/backend/document-buffer.ts`): the content-editing layer is a live canvas working copy over the store — read/write/edit + sync + diffs — not a message conduit and not a universal document abstraction, so "channel" is now reserved for real transports (IPC, the pane↔agent bus). The rename clarified a three-way layering that the earlier "content store, not filesystem" and "editor state vs versioning" sections only implied:

- **Document store** (substrate, shared): id→bytes + versions + opaque `meta`; path-unaware; the git backend slots in behind it later. Heavy use _outside_ the snapshot flow — the live `read`/`setCurrent` editing path and the `uix-resource://canvas.<workspace>/doc/...` render path dominate; snapshot/`getVersion` is the minority.
- **Buffer** (feature-owned): composes the store with the feature's own validate/format pre-store step (`canonicalizeHtml` for canvas) plus change detection; the store never calls feature code, and feature writes route through the buffer rather than reaching into the store directly. Name buffers after document purpose (`CanvasDocumentBuffer`), not just mechanisms like anchoring.
- **Anchoring** (reusable _text_ library, opt-in): not universal. A case-2 React/JSON state pane skips anchors entirely and brings schema-validate + stable-stringify, sharing only the document store — so it writes its own buffer, not a generalized `DocumentBuffer`. Whatever turns out genuinely shared between two buffers gets lifted then, with two real consumers, not guessed now.

Conduit inventory note: the cockpit↔agent channels (state-messages, turn-state, the transcript stream) are generalized substrate, but the pane↔agent **typed channel** AGENTS.md promises is still only the canvas-specific iframe trio (`uix-resource://canvas.<workspace>/doc/...` render down, the shim's `postMessage`→`uix:canvas-writeback` up, `uix:canvas-changed` invalidate). The "one channel API, two transports" invariant is half-built: only the iframe `postMessage` transport exists; the in-process transport a directly-mounted React pane needs does not. Generalizing that transport is the sibling of the document-store generalization — independent of it, and the canvas currently fuses both.

### 2026-06-13 — latest-vs-snapshot model and pre-user submit state

Settled the snapshot model that fixes the current "diffs only exist after the agent has touched the canvas" hole without turning every writeback into durable history. There are two stores conceptually: **latest** is the mutable working HTML file updated by every iframe writeback and agent canvas tool; **snapshots** are immutable versions containing content plus anchor state. A snapshot becomes part of the branch only when a UIX-private `uix.turn-state` `CustomEntry` points at it. The diff sent to the agent is derived, not authoritative: nearest upstream turn-state snapshot → newly-created submit snapshot, rendered with anchors into the hidden `uix.state` custom message.

Submit prep must finish **before** the user message is appended. The driver can use the raw user text to prepare reminders/context, but the session order should be: `uix.turn-state` custom entry, hidden model-visible `uix.state` custom message, then the user message. This restores the original tree-navigation invariant: when the human steps to the gap before a user message, the hidden state needed to hydrate that point is already upstream. The current `before_agent_start` state-message flush is therefore transitional for this use; the target is a UIX submit-prep phase that writes customs before `session.prompt(text)`.

Agent writes follow the same boundary discipline in the other direction. During a run, `canvas__anchor_write`/`edit` update latest and the live anchored channel, and their tool results tell the agent what it saw. At `agent_end`, if a canvas changed, UIX snapshots final latest and appends a post-run `uix.turn-state` pointer. The next user-submit diff then compares against this post-agent snapshot. This skips per-tool durable entries for the first cut while preserving the run-boundary stack needed for undo/preview; if crash recovery or fine-grained undo later requires it, writebacks/tool calls can create ephemeral git versions without making them branch-visible.

Developer preview diffs are a separate surface: a `uix:canvas-writeback` handler may peek latest against the nearest upstream snapshot and send an anchorless/human-facing preview to the agent pane, cleared on submit. Those previews do not commit snapshots and do not affect what the agent sees.

### 2026-06-12 — next-version canvas edit: base as observation frontier, snapshot commit, conflict guard, anchor-range reads

Worked the next version of the agent↔canvas edit surface, triggered by designing immediate (mid-run) state delivery and finding the `consume` model can't support it. The whole design collapses onto one primitive.

**`base` is the agent's observation frontier, and the branch proves it.** Today `#sync` computes `diff(live, store)` and immediately `reconcile`s it away, keeping no record of what the agent has actually seen — so the diff base is implicit and unrecoverable. Promote it to an explicit `base`: the document state the agent has **demonstrably observed**. It is not a single snapshot but a **frontier** advanced by every observation — a read covers its range, a `write`/`edit` covers its result, a delivered `uix.canvas-diff` envelope covers everything. Because reads, edits, and envelopes are all persisted (tool calls / branch entries), the frontier is branch-derivable: same property as the state-message latch ("the branch is the latch", [agent-state-messages](./agent-state-messages.md)), now applied to the channel itself. This one primitive serves three masters — change-only diff reporting, snapshot commit, and the edit conflict guard below.

**`consume` becomes snapshot create / commit (vaguely a chain).** Replacing "compute-and-advance in one step": **peek** computes `diff(base, current)` and pins the result as a _document fork_ (lines + the anchors minted for the diff + the anchor cursor) — not a content string, because the envelope the agent receives quotes those exact anchors and the committed state must own them. **commit** swaps the fork in as the live document; **cancel** drops the fork and keeps diffing against the unmoved `base`. There is no hunk buffer to merge — a diff is always _derived_ from (base, current), so cancel just recomputes from the unmoved base and the peeked hunks reappear fused with anything edited since. The steer tier (immediate delivery, [agent-state-messages](./agent-state-messages.md)) uses this directly: peek at send (base untouched), ship envelope + human message as paired steers, **commit on the durable-id persistence callback** (the identity module already exposes it — persistence is proof the envelope is on the branch and in the model's reading order), drop on abort. In-flight rule: no second unconfirmed peek of the same canvas, else it duplicates the first's hunks; pi's queue is FIFO so "hold until the first confirms" is trivial.

**Commit is fast-forward-only.** The agent can move the live document between peek and delivery (and `edit` syncs human store content in first, so an agent edit already absorbs the human's changes). If the live doc has advanced past the snapshot's parent, blind-swapping the fork in would roll back the agent's own work — so an advanced doc **orphans** the pending snapshot: discard it. Nothing is lost — the agent both received the envelope _and_ absorbed the human edits via its own sync (told twice, not zero). At fork time, reserve the fork's anchor index range in the original too, so an orphaned anchor name can never later be minted for a different line.

**Edit conflict guard (closes a live clobber hole).** The boundary text-guard only verifies the `start`/`end` anchors (`findMatchingLine`); the interior of `[start, end]` is sliced by index and replaced wholesale — so a human's interior edit, folded in by `#sync`, is then silently clobbered by the splice. This is a lost update, exactly what verbatim-match edit tools exist to prevent, and we should not allow it (the cockpit is sole writer; the human is the authority). The tool call carries no interior view, but `base` does: compute `diff(base, latest)`, and if any changed region intersects the edit's target range, **block** and return a _structured, recoverable_ result — a fresh read over the range, a note that the human edited inside it, "re-make the edit against this" — not a bare throw, so the agent re-plans in the same turn (the auto-readback shape from the 2026-06-04 entry, now driven by `base` instead of only the match-guard).

**A read advances the frontier over its range.** When the agent reads `[start, end]`, `#sync` has already returned current content for that span — the agent genuinely saw it — so fold those lines into `base`. Two payoffs: (1) kills false conflicts for read-then-edit over the same span; (2) suppresses redundant envelope reporting — a line the agent read should not be re-announced in the next turn's diff, and because the envelope diffs against the same `base` this falls out for free (solicited read and unsolicited diff stop double-covering). Edge: a read partially overlapping a human change advances the covered lines and leaves the rest pending, so the next envelope reports a partial hunk — truthful, occasionally a touch confusing, acceptable. Restart seeds `base` from the **last committed envelope on the branch** (not from `current`, which would hide edits made while the app was closed).

**Anchor-range reads (new tool capability).** Today `read(start, end)` is index-only. Agents read-before-write by nature, and with only index addressing they guess a window and re-read to correct it. Add an **anchor-bounded read**: given `start`/`end` anchors, walk the linked list between those nodes (reuse the `#collectRange` traversal `edit` already uses) and return the slice — O(range), no index arithmetic. The first read of a fresh document still has no anchors (index or whole-doc), but every read/edit/write result already returns anchors, so subsequent reads can target precisely in one shot. Optional boundary text-guard mirrors `edit` for staleness detection.

Net: `consume` → snapshot create/commit; `base` becomes an explicit branch-proven frontier serving diff/commit/conflict; the interior-clobber hole closes; reads both target by anchor and advance the frontier. None of this blocks the state-message substrate refactor; it's the channel's own next unit, gated only on introducing `base` as tracked state.

### 2026-06-09 — anchors ride the version; the human diff becomes a durable custom message

Two revisions out of the durable-identity design walk ([conversation-render-primitives](./conversation-render-primitives.md) log of the same date). **Anchor state homes in the version's commit meta**, not a session entry and not the 2026-06-04 sidecar cache: the anchor↔line map + allocation index are a function of the document's edit history up to a commit, so storing them with it makes restore atomic and necessarily consistent — checkout hands back content and anchors together, and the C3 turn-state pointer alone stitches turn → version → both. Consequence: a version id names a commit-like object `{contentRef, meta}`, not a bare content hash (identical contents can carry different anchor states); blobs stay content-addressed underneath. Droppability survives as the fallback, not the mechanism: meta absent → regenerate renumbered → match-guard rejects stale historical anchors → the agent re-reads — and that re-read's context cost is exactly why continuity is worth engineering. Restore granularity confirmed at **turn boundaries** (pi CLI's model), so per-modification versions exist in the store but pointers are per-turn.

**The human-writeback diff is promoted from ephemeral transform to durable entry**: a `uix.canvas-diff` `CustomMessageEntry` (content = anchored diff, details = structured hunks for a rich chat block, display for the human-facing strip) appended at the submit boundary before the user message — the agent sees a non-user message, history survives resume/branch-walk, and the user entry stays verbatim. A sibling `uix.pane-visibility` entry (change-only, latched in main, coalesced at the same boundary) tells the agent what the human is looking at when they speak. Both specified in [persistence-and-session-foundation](../plans/persistence-and-session-foundation.md) C3; the consuming-`collectChanges()` landmine (log 2026-06-06) still applies unchanged.

### 2026-06-01 — framing, anchor model, doc system

### 2026-06-02 — document-store docs, filesystem parity, rollback model

Refined the channel from "canvas file with edit tools" into "versioned document-store document with anchored tools." Pane content is addressed by document id and committed through the cockpit, with local `.uix/` storage as a hidden content-addressed object store; content hashes name immutable version objects, while line anchors remain assigned IDs. Normal filesystem `read`/`write`/`edit` should use the same anchored grammar via internal state and reconciliation, without model-visible versions; after a `bash`/editor/git mutation, `read(..., changes)` can reconcile against disk and return fresh anchored hunks. Pane tools probably also do not need model-supplied revisions: UIX tracks current checkout/commit internally, returns commit ids/results for observability, and rejects/conflict-reports when stale anchors intersect changed content. Conversation history and pane history are linked by pointers: each conversation node records current doc commits, stepping through the conversation graph previews panes, and rollback is one append-only graph interaction with conversation-only, pane-only, or both actions. Human pane changes are committed as hydrated document diffs before the next user turn and surfaced to the agent plus a collapsible diff area above the input.

### 2026-06-04 — future: optimistic concurrent editing over anchors

Deep optimization, deferred — recorded so the writeback model can grow into it. Because edits address lines by anchor, not position, an agent edit and a human edit to **mutually exclusive regions don't conflict, and we can detect when they do**. Treat agent+human as optimistic concurrency (not locking — nobody holds a lock): both write freely, conflicts are caught at edit time and the loser retries on top of the winner.

The foundation is already in place from the writeback work. Sync-before-edit silently merges non-colliding human edits into the live doc (no data loss); the match-guard rejects an agent edit whose anchor range intersects a human-changed line. The `AnchoredChange` hunks (old→new line mapping) turn that bare rejection into an **auto-readback**: on collision, find the hunk that replaced the agent's stale anchor and return its `newLines` (fresh anchors) in the failed edit's tool result, so the agent retries on top of the human's version with no manual re-read. This stays on the `customTools` path — the readback rides the edit tool's own result, no lower-level hook needed.

The refinement over today's turn-start surfacing is about **announcement timing, not merging**: while the agent is mid-sequence, keep merging non-colliding human edits silently and accumulate the touched regions, but don't interrupt the agent with them; only collisions surface immediately (the auto-readback). At the end of the agent's turn sequence (the `agent_end` signal) surface the accumulated human regions in one batch — via the next turn's `contextForTurn`, or a proactive follow-up. Granularity is line/block (anchor-level); two edits to the same line are a genuine collision that can't auto-merge.

### 2026-06-04 — writeback surfacing landed: reconcile + per-turn context injection

Built the human-writeback _surfacing_ half of U2 (the agent→pane half — tools + live refresh — was already in). Two findings reshaped the plan's "surface via tool-result" choice. First, a `changes` tool is the wrong shape: making the agent _ask_ for human edits is exactly the generic-FS-agent behavior UIX exists to beat — the thesis is that UIX context-engineers the turn. Second, the plan/design had collapsed two distinct surfaces into "rewrite the user turn = lower-level path (U3)," but there are **three**: (1) tool result — agent pulls, wrong; (2) **augment the outgoing prompt at `session.prompt(text)`** — pure context engineering, available _now_ on the high-level `customTools` API because the driver owns that string; (3) pi's `context` hook — per-LLM-call message transforms, genuinely needs `createAgentSessionFromServices` (U3). Surface #2 delivers what we want without the lower-level refactor.

Implementation: `DocumentChannel` gained `#sync` (canonicalize the store's current content, compare to the live doc, `reconcile` only on divergence, **no commit** — the content came from the store) and `collectChanges()` (sync every touched doc, return the per-doc human-diff). `#sync` is now intrinsic to `read` and `edit` — and on `edit` it's a **correctness guard, not a nicety**: an edit computed against a stale cache would re-commit the agent's whole view and silently revert a concurrent human edit to an untouched line (covered by a golden test). `AgentBinding` grew `contextForTurn?(): Promise<string|null>`; the driver concatenates each binding's block and prepends it to the prompt; the canvas binding renders the diff as a `<canvas-changes>` block with fresh anchors so the agent can act without a re-read. Honest boundary deferred to U3: **mid-turn** human edits (only the `context` hook fires between LLM calls — the clobber-guard still protects correctness, the agent just isn't _told_ until the next turn) and Half-B input dedup. This is a better-motivated U2/U3 line than "tool-result vs rewrite." Still open: the shim + transport that actually delivers a human edit into the store (the canvas iframe is sandboxed off `window.uix`) — open question #4, the next piece.

### 2026-06-04 — anchor-state resume cache, manual-edit reconcile

Worked the consequence of two things the current cut punts on: **session resume** and **branches**. Today `DocumentChannel` holds the anchor↔line map in memory only and regenerates it from content on load — fine now, but a reloaded session would reissue _fresh_ anchors, invalidating every `<anchor>§<text>` already in the transcript and forcing a full-document re-inject to keep editing. Cheap resume therefore needs the _exact_ prior anchor↔line binding restored, not just the content.

Decision: persist the editor state as a **droppable sidecar cache, not authority**. The canvas file stays plain HTML and authoritative (what `canvas://` renders and what a human edits); alongside the session/transcript we persist the `AnchoredLine[]` (anchor+text per line) plus `nextAnchorIndex` and a content hash, keyed with the session. This revises the "never persisted" line above to "never persisted _as authority_": the cache is an optimization to skip re-injection, and losing or invalidating it degrades to a re-read — never data loss — so it doesn't make the filesystem load-bearing.

Why serialize `Line[]` rather than raw-text-plus-cursor: the naïve "store plain text, regenerate anchors" loses the binding the transcript references, so a working resume cache must store anchor↔text per line anyway — i.e. it _is_ a `Line[]`. And why a cache beside the content rather than making the `Line[]` the stored content: doing the latter would put anchors into the rendered/human-edited artifact, force a strip/reattach on every serve, couple the generic `DocumentStore` to the anchored editor, and make the ephemeral-by-design thing authoritative — all against the standing principles.

This also surfaces a **second human-edit channel** the synthesis above didn't cover. In-pane edits arrive over internal eventing (no watcher, no echo problem). But the plan was agent-only and the user expects to eventually hand-edit the canvas file _on disk_ — an out-of-band change that does ride `fs.watch`, reintroducing echo-suppression. Resolution: reconcile on **startup and external fs events**, gated on a content-hash compare so the common case (hash match → rebuild list from the snapshot) is nearly free and the Myers `reconcile()` (<100ms, already in `AnchoredDocument`) runs only on real divergence; exclude the channel's own commits from the watcher (suppress across own write, or skip when the post-write hash matches what was just written) — the same self-write shape the pane writeback already handles. All deferred; the in-memory-only map is accepted for now.

### 2026-06-02 — git backend, editor/versioning split, value-first build units

Resolved the versioning backend and used it to split this thread for sequencing. **Git is the backend for both stores** (detail now lives in [pane-and-file-versioning](./pane-and-file-versioning.md)): the owned `.uix` pane store is a bare repo driven by plumbing (commits + refs, not stash, not porcelain) — git shas fill the content-hash/object-id role while line anchors stay assigned IDs; branching is emergent from commit parents (we don't mirror pi's conversation tree with refs), and since pi never deletes conversation branches, `gc.auto=0` keeps history. The opt-in user-file store is `git stash create`-style snapshots built via a scratch `GIT_INDEX_FILE` → `write-tree` → `commit-tree` so they capture untracked files non-destructively, **per run** (snapshot at user-submit + at rollback-initiation, not per agent turn), with async app-managed labels that promote a snapshot to a durable ref, and restore made non-destructive by construction (snapshot-then-apply). The two stores are wholly separate implementations sharing only the git backend and a small conversation-node meta slot `{panes, userSnapshot?}`.

The sequencing key was separating three things previously fused: the anchored **editor** (regenerable, session-scoped, no persisted state) vs **versioning** (durable, behind the document-store `getCurrent`/`setCurrent`/snapshot seam) vs the **channel** (current content + diff). Because the anchor map regenerates from content, versioning slots behind the store interface and rollback restores content rather than editor state — so versioning can land late. Re-derived the work into capability units P0–U7 and chose **value-first** ordering: ship the anchored core + live canvas channel (P0–U2) on `customTools` first to prove the integration, then the lower-level pi refactor (U3), filesystem parity (U4), and the versioning/rollback units (U5–U7). Surfacing the human-diff via the tool-result channel (not a rewritten user turn) is what keeps U2 on `customTools`. Spun the versioning reasoning into its own design note and promoted the P0–U2 plan.

### 2026-06-08 — shim scope: form/control writeback, not global edit mode

The first writeback shim made the whole canvas body `contenteditable`, which proved the wrong default: it let humans select, edit, and delete arbitrary authored elements. ### 2026-06-06 — landmine: the pending-diff UI must not call `collectChanges()`

Surfaced while planning the conversation pending-diff strip; recording because it is invisible from pi and easy to get wrong in our own code. **`collectChanges()` is consuming.** `#sync` runs `reconcile(canonical)`, which advances the diff base and reassigns anchors, so once it runs the live doc equals the store and a second call returns `[]`. The single _consuming_ read belongs to the turn-context surface — `contextForTurn` today, the `input`-hook transform after the C1 extension promotion ([persistence-and-session-foundation](../plans/persistence-and-session-foundation.md)) — fired once at submit. Any UI that shows the human's pending edits _before_ submit must read them with a **non-mutating peek**: canonicalize the store's current content and Myers-diff it against the live `doc.read()` _without_ applying/reconciling. It must not call `collectChanges`/`read`/`edit`, which all `#sync`. Reuse `collectChanges` for display and it silently consumes the diff — the agent simply stops being told about human edits. Add a read-only `previewChanges(docId)` to `DocumentChannel` when the pending-diff UI lands.
