# Current state

The current state of the system: what's built, how the code is shaped, and the open questions we haven't resolved. Living architecture docs always describe HEAD and are rewritten freely to match the code. This is the dev-facing architecture-of-record; the vision lives in the repo-root [`../../AGENTS.md`](../../AGENTS.md), and the user-facing substrate reference that ships lives in [`../../src/docs/`](../../src/docs/).

**Built:**

- Electron + electron-vite scaffold (`electron.vite.config.ts`, `src/main`, `src/preload`, `src/renderer`).
- Typed IPC scaffold (`src/shared/ipc.ts`, preload bridge).
- Pi `createAgentSession` driver in the main process (`src/main/agent/driver.ts`).
- Lifetime-scoped disposables and lifecycle helpers (`src/main/lifecycle.ts`). See [conventions](./conventions.md).
- Chat feature (`src/features/chat/`) — pure user space: an entry file contributing one surface over the substrate-owned agent channels (`@uix/api/agent-channels`), default chat blocks, exact first-party canvas tool block renderers, chat CSS as a native CSS module script, and a local Iosevka font served through the surface pipeline's files route.
- **Single-page workspace renderer** (`src/renderer/workspace/`) — one page is one workspace; multi-workspace is multi-BrowserWindow; no Host frame, iframe, or postMessage bridge (collapsed in `8adeb7d` — see [workspace-runtime-foundation](../plans/archive/workspace-runtime-foundation.md)). The surface list is registry-driven: `useSurfaces()` fetches `uix.surfaces` and re-fetches on `surfaces_changed`; every surface — chat and canvas included — is dynamic-imported from its content-hash-busted substrate-origin URL, validated (`validateSurfaceContribution`, mirroring the loader's definition validation), and mounted behind a per-surface error boundary (a failing surface renders an error card, the frontend twin of the loader's `failed[]`). `SurfaceMount` mints each surface's typed channel client from its contract and adopts/unadopts its CSS module sheets while mounted. `WorkspaceClient` wraps the preload `ChannelTransport` (`window.channels`); the client React context lives in `@uix/api/workspace`, so feature components share it through the shared-module instance.
- **Typed channel substrate** — features declare schema-only `ChannelContract`s in `shared/`, each carrying its owner channel id (`feature`, stated once where the contract is defined and checked loudly at handler registration and publisher minting); the backend merges executable handlers via `withHandlers` (requests/responses parsed both directions by `ChannelRegistry`, `src/main/channels/registry.ts`); the frontend derives fully typed clients via `createChannelClient` (`@uix/api/workspace`, owner read from the contract, events validated on subscribe); backend feature code publishes through contract-derived typed publishers minted per feature by `FeatureEventPublisherFactory` — presenting a contract is the only way to obtain publish capability.
- **Feature contribution system** (`src/main/features/`) — `FeatureDefinition` (`id`, optional `context(ctx)` hook, `contribute(ctx)`) feeding facet registries for resources, channels, agent tools, turn state, agent context, and surfaces (`surfaces: string[]` frontend entry-file refs resolved against the feature entry's dir; each module default-exports its `defineSurface` result); `registerFeatureContributions` bags each facet registration per feature. There is no compiled-in feature inventory — chat and canvas are ordinary manifest entries loaded from `src/features/` source. Substrate agent channels (prompt/history/events) register under the reserved `agent` id; the surface list/changed channel under `uix`.
- **Feature loader** (`src/main/features/manifest.ts`, `loader.ts`) — one load pass activates the manifest's entries in manifest order. `uix.workspace.json` (`name` + ordered feature objects `{ entry, settings }`, with entry refs relative to the manifest or absolute) is the sole composition record; each jiti-loaded entry is validated as a default-exported `FeatureDefinition` (shape, id grammar, reserved `agent`/`uix` ids, duplicate ids, setting definitions with explicit defaults), settings are hydrated before `context()`, and contributions register through `registerFeatureContributions` under a per-feature bag in `featuresBag`. Feature code value-imports the blessed backend set through the loader's alias table: `@uix/api` prefix-mapped onto the implementation dir the composition root supplies (`FeatureSubstrate.apiModuleDir` — the API is self-contained, so one directory serves it) plus exact typebox entries (its package has an `exports` map but no `main`); everything else resolves by node walk-up from the feature file, so `#shared`/`#backend` are unreachable. `/reload` re-reads the manifest/workspace settings and re-runs everything with fresh context/bags, then delegates to pi; a throwing feature lands in `failed[]` without aborting the pass, while a malformed manifest rejects the pass before anything is cleared (startup boots with no features and logs loudly so the pilot can fix it and `/reload`). ([workspace-manifest-not-discovery](../decisions/2026-07-02-workspace-manifest-not-discovery.md), [features-are-the-loadable-unit](../decisions/2026-07-01-features-are-the-loadable-unit.md), [runtime-surface-pipeline](../decisions/2026-07-02-runtime-surface-pipeline.md).)
- **Workspace settings** (`src/main/workspace-settings.ts`, `src/api/settings.ts`) — feature definitions may declare TypeBox setting schemas with explicit defaults before `context()` runs. The loader binds settings to the manifest entry being activated, hydrates missing values into that entry's `settings` object, rejects unknown persisted setting keys, and injects validated `ctx.settings` into backend feature code. Plain object defaults merge recursively; arrays, scalars, and `null` are atomic. Settings are memory-authoritative while running, fire same-process `onChange` handlers synchronously after persistence is scheduled, and flush dirty feature settings back to `uix.workspace.json` atomically on a debounce. There is no live manifest watcher; external/agent edits are picked up by `/reload`.
- **Surface module pipeline** (`src/main/features/surface-pipeline.ts`) — esbuild bundles each registered surface entry on demand (feature-local code inlined; on-disk tsconfigs never read, so repo aliases can't leak in). The blessed shared set (`src/shared/surface-shared-modules.ts`: react, typebox, `@uix/api/workspace`/`agent-channels`/`resources`) maps to virtual CommonJS modules reading the page-populated global (`src/renderer/workspace/provide-shared-modules.ts`), so every surface shares the page's instances; CSS module scripts (`with { type: "css" }` — required, checked at bundle time) stay external with content-hash-busted files-route URLs. Modules serve at `uix-resource://uix.<ws>/surface/<feature>/<i>.js?v=<hash>`; feature dirs serve read-only under `/surface-files/…` (fonts, assets). The page CSP allows scripts/styles/fonts from exactly that origin; the scheme is `corsEnabled` and the routes echo non-`uix-resource` request origins (module scripts are always CORS-mode and the page is a different origin — dev server or `file:`) while refusing `uix-resource://` origins, so canvas iframes cannot fetch modules.
- **`@uix/api` is self-contained; `src/shared/` is cockpit-internal only** — `src/api/` holds everything feature-facing plus its own implementation (including `resource-routes`, `channel-normalization`, `contribution-id`, and the `agent-channels` contract with the transcript shapes); `src/shared/` holds only cross-process cockpit contracts (`ipc.ts` transport constants/picker types/`uixChannels`, the surface shared-module list). Nothing re-exports across that line.
- **App shell + start picker** — startup splits into shell (process handlers, quit semantics) and `openWorkspace(target)` (state root, registries, driver, feature load, window). The workspace target is `UIX_WORKSPACE` (manifest path or dir), else a cwd holding a manifest (repo dev flow), else the start picker (`src/renderer/picker/`): a small shell window over `uix:picker:*` channels offering recents (capped newest-first JSON in `userData`, pruned of deleted manifests — `src/main/recents.ts`) and create-new (native dir dialog; scaffolds editable copies of the default features and writes the manifest; a folder already holding a manifest is adopted, not overwritten). `resolveWorkspace(target)` derives `stateRoot`/default `agentCwd` from the manifest's directory. One open workspace per App instance; concurrent multi-workspace windows wait on per-workspace substrate scoping.
- **Resource routes substrate** — a single `uix-resource://` custom protocol with route-based dispatch replaces per-resource scheme registration. `src/api/resource-routes.ts` defines the route model (`path`, `query`, `origin`), normalization, branded `ResourceUrl` encoding, and URL decoding. `src/main/resources/registry.ts` stores `NormalizedResourceRoute` registrations in a Map and dispatches incoming requests by decoding URLs against each route pattern, handing pre-parsed `{ params, query }` to feature handlers via `ResourceRequestContext`. Feature authors declare routes with `createResourceAddressBuilder` from `@uix/api/resources`, which returns `{ route, url(), origin() }` — `route` goes into the contribution, `url()`/`origin()` are consumed by renderer code. The builder is a transitional hybrid; it dissolves into substrate-owned transport when surface contributions land. See [resource-routes](../plans/archive/resource-routes.md).
- Canvas surface: feature-origin `uix-resource://canvas.<workspace>/doc/...` resource route (declared by `createResourceAddressBuilder`), key-addressed mutable latest store, immutable JSON canvas snapshots with anchor-state meta, private turn-state contributions for per-submit and post-agent `uix.turn-state` snapshot refs, snapshot-derived `<canvas.canvas-diff>` agent context, anchored canvas tools, whole-document refresh, and an injected writeback shim for form/control and explicit `contenteditable` edits (`src/features/canvas/backend/`, `src/features/canvas/workspace/`, `src/features/canvas/shared/`, `src/main/turn-state/`, `src/main/agent-context/`). Writeback postMessage listener still hand-rolls origin/source/key checks (it forwards into the typed channel client) — substrate-owned message routing for nested iframes remains deferred until the iframe surface transport exists. See [canvas-stage-one](../decisions/2026-05-31-canvas-stage-one.md), [canvas-data-channel](../design/canvas-data-channel.md), and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

**In flight:** turn state (`src/main/turn-state/`) owns appending opaque per-feature refs at run boundaries and agent context (`src/main/agent-context/`) materializes model-visible `<uix-state>` sections from them; the turn-state contract still needs the navigation half — preview/restore callbacks beside prepare — which unblocks branch preview/restore and folds in anchor rehydration ([persistence-and-session-foundation](../plans/persistence-and-session-foundation.md)). Durable transcript identity ([durable-transcript-identity](../plans/durable-transcript-identity.md)) is the next substrate needed before interactive or stateful chat blocks. Canvas edits its working copy through `CanvasDocumentBuffer` (a purpose-owned working copy over the store, not a transport; "channel" names only message conduits). Filesystem-tool parity and surface versioning/rollback are tracked in [canvas-data-channel](../design/canvas-data-channel.md) and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

**Deferred from resource routes:** two items captured in [open questions](./open-questions.md) — substrate-owned postMessage origin/source filtering, and the `createResourceAddressBuilder` dissolving when surface contributions land (the builder's `url()` and `origin()` become substrate-internal at that point).

What's _next_ lives in the plans backlog ([`../plans/backlog.md`](../plans/backlog.md)); per-decision history is in [`../decisions/`](../decisions/).
