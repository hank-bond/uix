# Current state

The current state of the system: what's built, how the code is shaped, and the open questions we haven't resolved. Living architecture docs always describe HEAD and are rewritten freely to match the code. This is the dev-facing architecture-of-record; the vision lives in the repo-root [`../../AGENTS.md`](../../AGENTS.md), and the user-facing substrate reference that ships lives in [`../../src/docs/`](../../src/docs/).

**Built:**

- Electron + electron-vite scaffold (`electron.vite.config.ts`, `src/main`, `src/preload`, `src/renderer`).
- Typed IPC scaffold (`src/shared/ipc.ts`, preload bridge).
- Pi driver in the main process (`src/main/agent/driver.ts`). Electron computes one app-owned Pi profile at `<userData>/pi` and injects it into every workspace driver; the host Pi profile is never inherited, while each workspace cwd continues to supply project-local `.pi` settings/resources and sessions remain in the workspace's `.uix/sessions`. An eager auth-free `SessionManager` opens the durable id in the substrate-owned `session.selected` workspace setting, falls back to the newest graph when that selection is stale, and keeps initial history independent from live agent startup; session summaries project Pi's explicit title independently from a 512-code-point first-user-message preview, leaving empty-session fallback copy to consuming features; the first prompt or session mutation lazily creates Pi's supported `AgentSessionRuntime`. One pre-runtime `AgentSessionServices` tier loads `AuthStorage`, `ModelRegistry`, settings, and Pi resources/extensions whenever model/auth availability needs them, so extension-registered providers are visible before a prompt; initial runtime creation reuses that exact tier, while replacement runtime generations recreate cwd-bound services and rebind transcript observation/subscriptions to the active `AgentSession`. `session_history({ sessionId? })` returns one graph's transcript plus authoritative `SessionSummary` without activating an explicit non-selected target. `list_session_summaries({ limit })` selects recent session files newest-mtime-first, then sequentially reads only summary-bearing records from that bounded set without opening Pi services. `set_session_title({ sessionId, title })` sets or clears native append-only Pi title metadata on any graph without switching or opening Pi services, remains valid during an active run, and returns the updated summary. The payload-free `new_session` request rejects during an active run, commits current feature turn state, replaces the graph through the runtime, waits for fresh-state restoration, persists the replacement selection, and returns its `SessionSummary`. `/reload` recreates an already-used pre-runtime services tier, delegates to Pi's native reload once a runtime exists, and initializes neither solely for reload. `listModels()`/`setModelFavorite()`/`status()`/`selectModel()` answer before the first prompt; the model catalog decorates each available model with workspace-local favorite status, favorite updates persist idempotently under `agent.favoriteModels` while retaining unavailable references for reconnects, and `selectModel` validates against Pi's available models, persists the workspace default (`agent.defaultModel`), and switches a live session via `session.setModel` (native `model_change` state). At session open the workspace default applies only when the branch carries no `model_change` of its own; with neither, Pi's own resolution runs and status reports an explicit no-model-chosen state. A `model_select` installer mirrors every live Pi model change into status, emitted as `status_changed`. The provider-auth catalog merges model and registered OAuth providers into `ProviderAuthCatalogEntry` values with generic methods and may expose the active environment-variable name, command-backed source/location classification without raw command text, and only the last four characters of a sufficiently long environment-backed or stored literal API key as non-secret replacement-form context. Chat explains that replacement writes a literal `auth.json` key which takes precedence without mutating external sources; complete values and command/OAuth details never cross. The credential-save channel validates against that live catalog, checks required fields, and persists generic API keys through Pi with complete request redaction. The driver-owned OAuth coordinator (`src/main/agent/auth-flow.ts`) exposes one correlated callback flow through sensitive-log-described agent channels; Electron injects `shell.openExternal`, Pi alone persists credentials, and either successful path refreshes model availability.
- Lifetime-scoped disposables and lifecycle helpers (`src/main/lifecycle.ts`). See [conventions](./conventions.md).
- Chat feature (`src/features/chat/`) — pure user space: an entry file contributing one surface over the substrate-owned agent channels (`@uix/api/agent-channels`), default chat blocks, a status-bar recent-conversation pill + picker with inline explicit-title editing over the workspace session capability (`SessionPill.tsx`), a model pill + searchable Favorites / All models picker over `list_models`/`set_model_favorite`/`agent_status`/`select_model`/`status_changed` (`ModelPill.tsx`) — both pickers share filled/empty circle current-selection markers — and a provider modal whose API methods toggle descriptor-derived inline forms while subscription methods drive one restorable OAuth state machine (catalog-provided start actions, automatic system-browser authorization/device-code opening, correlated prompt/select answers, progress, cancel/retry/success), confirm connections in place, and explicitly hand off to a provider-seeded ordinary model search. It also supplies exact first-party canvas tool block renderers, component-owned and narrowly shared Chat styles as explicitly ordered native CSS module scripts (each structurally `@scope`-wrapped at adoption, with semantic local tokens, accessible focus treatment, and reduced-motion-safe confirmation), and a local Iosevka font served through the surface pipeline's files route.
- **Single-page workspace renderer** (`src/renderer/workspace/`) — one page is one workspace; multi-workspace is multi-BrowserWindow; no Host frame, iframe, or postMessage bridge (collapsed in `8adeb7d` — see [workspace-runtime-foundation](../plans/archive/workspace-runtime-foundation.md)). The surface list is registry-driven: `useSurfaces()` fetches `uix.surfaces` and re-fetches on `surfaces_changed`; every surface — chat and canvas included — is dynamic-imported from its content-hash-busted substrate-origin URL, validated (`validateSurfaceContribution`, mirroring the loader's definition validation), and mounted behind a per-surface error boundary (a failing surface renders an error card, the frontend twin of the loader's `failed[]`). `SurfaceMount` mints each surface's typed channel client from its contract and adopts/unadopts its CSS module sheets while mounted. `WorkspaceClient` wraps the preload `ChannelTransport` (`window.channels`); the client React context lives in `@uix/api/workspace`, so feature components share it through the shared-module instance. The page also owns one `ActionRegistry`: `SurfaceMount` binds registration to the owning feature id, `useActionContribution` owns register/update/unregister lifetime, and `useActionCatalog`/`useInvokeAction` expose only flat catalog entries and id-based invocation; Chat's mounted model pill contributes `chat.models.favorites` and `chat.models.all` as the first proof. Normalization splits each authored leaf into its executable registration, serializable `ActionCatalogEntry`, and canonical default-binding template entry; the registry maintains and publishes independent snapshots of the catalog and aggregate default template, so enabled/running-only updates never request durable-default reconciliation. `KeybindingSync` binds that template to the reserved `uix` channel: it subscribes before initial reconciliation, batches same-turn template changes, retries after surface/reload notifications, ignores stale request responses after a newer published confirmation, and leaves the registry's confirmed binding snapshot explicitly `undefined` until main responds. Once hydrated, the registry joins shortcut/`null` values onto active catalog entries, resolves `mod` for the renderer platform, marks every active conflict claimant, and separately derives an unresolved projection for persisted ids with no live action; removing and reinstalling actions reclassifies those durable entries without changing them. One bubble-phase workspace dispatcher translates browser logical keys and produced Shift/Option characters back to persisted gestures, then invokes unique confirmed claimants through the registry. Local/modal handlers get first refusal; composition and AltGraph are untouched; claimed repeats, conflicts, disabled actions, and already-running actions suppress browser defaults without another callback. Ctrl/Command/Alt bindings remain global in editable controls while Shift-only gestures do not. Keyboard callback failures publish registry diagnostics and enter the browser's report-error path. The renderer also owns one workspace session controller: it publishes active and recent session projections read-only to surfaces, hydrates them independently with state/request-version stale-result guards, observes agent activity independently from Chat, refreshes recents after successful mutations and completed runs, reconciles the active first-message preview from post-run summaries, exposes busy-aware switching, serializes title changes with other session mutations while allowing them during agent activity, updates title responses without changing selection, and backs the substrate-owned `uix.session.new` action whose materialized `mod+n` binding skips the request while busy.
- **Typed channel substrate** — features declare schema-only `ChannelContract`s in `shared/`, each carrying its owner channel id (`feature`, stated once where the contract is defined and checked loudly at handler registration and publisher minting); the backend merges executable handlers via `withHandlers` (requests/responses parsed both directions by `ChannelRegistry`, `src/main/channels/registry.ts`); the frontend derives fully typed clients via `createChannelClient` (`@uix/api/workspace`, owner read from the contract, events validated on subscribe); backend feature code publishes through contract-derived typed publishers minted per feature by `FeatureEventPublisherFactory` — presenting a contract is the only way to obtain publish capability. Request, response, and event descriptors can replace sensitive payloads with redacted descriptions in both terminal and NDJSON crossing logs; raw logging remains the default.
- **Feature contribution system** (`src/main/features/`) — authors use `defineFeature(...)` to preserve an optional settings schema through typed `context(ctx)`/`contribute(ctx)` callbacks while producing the plain runtime `FeatureDefinition` (`id`, optional `context(ctx)` hook, `contribute(ctx)`) that feeds facet registries for resources, channels, agent tools, Agent system-prompt sections, Pi skills, turn state, agent context, and surfaces (`surfaces: string[]` frontend entry-file refs resolved against the feature entry's dir; each module default-exports its `defineSurface` result); `registerFeatureContributions` bags each facet registration per feature. A single internal system-prompt assembler snapshots feature system-prompt sections plus generated agent-context vocabulary when the Pi runtime starts/reloads; feature-relative skill paths reach Pi through one `resources_discover` installer. There is no compiled-in feature inventory — chat and canvas are ordinary manifest entries loaded from `src/features/` source. Substrate agent channels (`prompt`/`session_history`/`list_session_summaries`/`new_session`/`switch_session`/`set_session_title`/events, plus `list_models`/`set_model_favorite`/`agent_status`/`select_model` and the `status_changed` event) register under the reserved `agent` id; the surface list/changed channel under `uix`.
- **Feature loader** (`src/main/features/manifest.ts`, `loader.ts`) — one load pass activates the manifest's entries in manifest order. `uix.workspace.json` (`name` + ordered feature objects `{ entry, settings }`, with entry refs relative to the manifest or absolute) is the sole composition record; each jiti-loaded entry is validated as a default-exported `FeatureDefinition` (shape, id grammar, reserved `agent`/`uix` ids, duplicate ids, setting definitions), settings are hydrated before `context()`, and contributions register through `registerFeatureContributions` under a per-feature bag in `featuresBag`. Feature code value-imports the blessed backend set through the loader's alias table: `@uix/api` prefix-mapped onto the implementation dir the composition root supplies (`FeatureSubstrate.apiModuleDir` — the API is self-contained, so one directory serves it) plus exact typebox entries (its package has an `exports` map but no `main`); everything else resolves by node walk-up from the feature file, so `#shared`/`#backend` are unreachable. `/reload` re-reads the manifest/workspace settings and re-runs everything with fresh context/bags, then delegates to pi; a throwing feature lands in `failed[]` without aborting the pass, while a malformed manifest rejects the pass before anything is cleared (startup boots with no features and logs loudly so the pilot can fix it and `/reload`). ([workspace-manifest-not-discovery](../decisions/2026-07-02-workspace-manifest-not-discovery.md), [features-are-the-loadable-unit](../decisions/2026-07-01-features-are-the-loadable-unit.md), [runtime-surface-pipeline](../decisions/2026-07-02-runtime-surface-pipeline.md).)
- **Settings substrate** (`src/main/workspace-manifest-store.ts`, `src/main/settings-registry.ts`, `src/main/workspace-settings-namespace.ts`, `src/main/workspace-settings.ts`, `src/api/settings.ts`) — persistence, schema logic, and live state are separate pieces composed by a facade. `WorkspaceManifestStore` owns `uix.workspace.json` as staged/active/stale generations: `stageFromDisk()` reads and structurally validates one detached generation, generation-bound locations expose `{ read, write }`, and `promote()` atomically makes the staged tree active while stale locations reject writes. Each generation retains its disk baseline; active writes mark the store dirty only on JSON change, flush rechecks the whole tree against that baseline, and changed content reaches disk through a debounced atomic replacement with retry-on-failure. `createWorkspaceSettings.reload()` parses composition and hydrates every registered workspace namespace against that same staged generation, rejecting malformed/unknown namespaces before promotion and returning the accepted feature refs to the loader, so composition and settings never come from separate reads. Every feature scope has one `defineSettings({ schema, default? })` definition for its complete object, while each substrate workspace scope wraps the same definition with a stable id through `defineWorkspaceSettingsNamespace(...)`: `Type.Object` gives named keys, `Type.Record` gives dynamic validated keys, and the helpers close either schema against unknown properties. The namespace descriptor is both registration input and the token passed to `forNamespace(...)`, whose owner-scoped handle derives key/value and complete snapshot/replacement types from that schema. The composition root currently registers named `agent` and `session` scopes plus the dynamic `keybindings` scope; `session.selected` stores only the selected durable session id, while keybindings map canonical action ids to one portable shortcut or `null` and explicitly default to `{}`. `hydrateSettings()` recursively fills whole-object defaults and validates the completed object; defaults materialize into the staged/live manifest rather than acting as a runtime overlay. `SettingsRegistry` owns live working state and routing over complete scopes: feature ids and workspace namespaces share one unprefixed id space, every registration starts provisional and returns an identity-aware commit/disposal token, every keyed `set` clones and validates the complete candidate scope, and trusted composition-root code can use its registered namespace descriptor to get a typed snapshot of or atomically replace a whole validated scope with one persistence write before changed-key notifications. Before commit, feature-local reads/writes/listeners work but persistence and app-wide notifications stay detached; commit writes the final object once and switches later sets to write-through. Workspace namespaces commit immediately after candidate adoption. Feature settings register into the feature bag before `context()`, commit only after every facet succeeds, and disappear without manifest materialization on failure; later bag disposal removes only the live scope registration, never its durable values. `ctx.settings`, substrate namespace handles, and `useFeatureSetting(featureSettings, key)` remain owner-scoped, while external/agent file edits arrive only through reload. The main keybinding request handlers are bound by namespace id to the reload-stable settings registry and reserved `uix` publisher; they normalize portable shortcuts, materialize missing renderer-declared defaults while persisted shortcuts and `null` win, replace complete candidates, and publish one confirmed snapshot when durable bindings change.
- **Surface module pipeline** (`src/main/features/surface-pipeline.ts`) — esbuild bundles each registered surface entry on demand (feature-local code inlined; on-disk tsconfigs never read, so repo aliases can't leak in). The blessed shared set (`src/shared/surface-shared-modules.ts`: react, typebox, `@uix/api/workspace`/`agent-channels`/`resources`) maps to virtual CommonJS modules reading the page-populated global (`src/renderer/workspace/provide-shared-modules.ts`), so every surface shares the page's instances; CSS module scripts (`with { type: "css" }` — required, checked at bundle time) stay external with content-hash-busted files-route URLs. Modules serve at `uix-resource://uix.<ws>/surface/<feature>/<i>.js?v=<hash>`; feature dirs serve read-only under `/surface-files/…` (fonts, assets). The page CSP allows scripts/styles/fonts from exactly that origin; the scheme is `corsEnabled` and the routes echo non-`uix-resource` request origins (module scripts are always CORS-mode and the page is a different origin — dev server or `file:`) while refusing `uix-resource://` origins, so canvas iframes cannot fetch modules.
- **`@uix/api` is self-contained; `src/shared/` is cockpit-internal only** — `src/api/` holds everything feature-facing plus its own implementation (including `resource-routes`, `channel-normalization`, `contribution-id`, and the `agent-channels` contract with the transcript shapes); `src/shared/` holds only cross-process cockpit contracts (`ipc.ts` transport constants/picker types/`uixChannels`, the surface shared-module list). Nothing re-exports across that line.
- **App shell + start picker** — startup splits into shell (process handlers, quit semantics) and `openWorkspace(target)` (state root, registries, driver, feature load, window). The workspace target is `UIX_WORKSPACE` (manifest path or dir), else a cwd holding a manifest (repo dev flow), else the start picker (`src/renderer/picker/`): a small shell window over `uix:picker:*` channels offering recents (capped newest-first JSON in `userData`, pruned of deleted manifests — `src/main/recents.ts`) and create-new (native dir dialog; scaffolds editable copies of the default features and writes the manifest; a folder already holding a manifest is adopted, not overwritten). `resolveWorkspace(target)` derives `stateRoot`/default `agentCwd` from the manifest's directory. One open workspace per App instance; concurrent multi-workspace windows wait on per-workspace substrate scoping.
- **Resource routes substrate** — a single `uix-resource://` custom protocol with route-based dispatch replaces per-resource scheme registration. `src/api/resource-routes.ts` defines the route model (`path`, `query`, `origin`), normalization, branded `ResourceUrl` encoding, and URL decoding. `src/main/resources/registry.ts` stores `NormalizedResourceRoute` registrations in a Map and dispatches incoming requests by decoding URLs against each route pattern, handing pre-parsed `{ params, query }` to feature handlers via `ResourceRequestContext`. Feature authors declare routes with `createResourceAddressBuilder` from `@uix/api/resources`, which returns `{ route, url(), origin() }` — `route` goes into the contribution, `url()`/`origin()` are consumed by renderer code. The builder is a transitional hybrid; it dissolves into substrate-owned transport when surface contributions land. See [resource-routes](../plans/archive/resource-routes.md).
- Canvas surface: feature-origin `uix-resource://canvas.<workspace>/doc/...` resource route (declared by `createResourceAddressBuilder`), key-addressed mutable latest store, immutable JSON canvas snapshots with anchor-state meta, a schema-bound `canvas.documents` turn-state cell for complete snapshot refs, snapshot-derived `<canvas.canvas-diff>` agent context, anchored canvas tools, a stable authoring system-prompt section plus an on-demand `canvas-authoring` skill, whole-document refresh, and an injected writeback/prompt shim for form/control and explicit `contenteditable` edits plus trusted-click `data-uix-prompt` Agent turns (`src/features/canvas/backend/`, `src/features/canvas/workspace/`, `src/features/canvas/shared/`, `src/main/turn-state/`, `src/main/agent-context/`). The cell restores referenced content plus exact anchor metadata and resets loaded working documents omitted by the target state through the selected-session activation and serialized reload paths. Writeback postMessage listener still hand-rolls origin/source/key checks (it forwards into the typed channel client) — substrate-owned message routing for nested iframes remains deferred until the iframe surface transport exists. See [canvas-stage-one](../decisions/2026-05-31-canvas-stage-one.md), [canvas-data-channel](../design/canvas-data-channel.md), and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

**In flight:** turn state (`src/main/turn-state/`, with selected-session commit/restore coordination in `src/main/agent/turn-state-lifecycle.ts`) now registers independently changing cells under feature-and-cell identities, validates complete plain-JSON snapshots against each cell's TypeBox schema, and commits only changed cells at run boundaries. The same cell contract carries `restore(latest | undefined)`; Canvas implements restoration of document content plus anchor metadata, while the selected-branch projection now derives transcript plus turn state as of the leaf and the restore scheduler validates projected values, restores features concurrently, and restores each feature's cells in declaration order; startup, replacement-session activation, and serialized feature reload invoke that scheduler before completion; reload publishes replacement surfaces only after restoration settles ([persistence-and-session-foundation](../plans/persistence-and-session-foundation.md), [session-history-and-switching](../plans/session-history-and-switching.md)). Agent context (`src/main/agent-context/`) continues to materialize model-visible `<uix-state>` sections from cell-scoped commit histories. Durable transcript identity ([durable-transcript-identity](../plans/durable-transcript-identity.md)) now shares the branch projection; New Session exercises the replacement restore lifecycle, while durable block-state joining remains. Recent summary discovery, explicit history reads, persisted selected-session identity, guarded renderer hydration/switching, and Chat's recent-session picker have landed; summaries expose explicit titles and bounded first-message previews independently, while renderer coordination and Chat's inline title editor complete the title slice. Canvas edits its working copy through `CanvasDocumentBuffer` (a purpose-owned working copy over the store, not a transport; "channel" names only message conduits). Filesystem-tool parity and surface versioning/rollback are tracked in [canvas-data-channel](../design/canvas-data-channel.md) and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

Feature activation now achieves substrate-owned per-feature atomicity: the provisional settings scope and every facet registration are enrolled in the same feature bag, grouped registration helpers clean up partial acquisition before rethrowing, and a failed feature leaves neither a live scope nor an earlier facet contribution while siblings continue. Persisted workspace candidate adoption is the preceding owner-level transaction; arbitrary side effects performed directly by trusted feature code remain outside substrate rollback, per [atomic candidates and feature activation](../decisions/2026-07-13-atomic-candidates-and-feature-activation.md).

**Deferred from resource routes:** two items captured in [open questions](./open-questions.md) — substrate-owned postMessage origin/source filtering, and the `createResourceAddressBuilder` dissolving when surface contributions land (the builder's `url()` and `origin()` become substrate-internal at that point).
