# Current state

The current state of the system: what's built, how the code is shaped, and the open questions we haven't resolved. Living architecture docs always describe HEAD and are rewritten freely to match the code. This is the dev-facing architecture-of-record; the vision lives in the repo-root [`../../AGENTS.md`](../../AGENTS.md), and the user-facing substrate reference that ships lives in [`../../src/docs/`](../../src/docs/).

**Built:**

- Electron + electron-vite scaffold (`electron.vite.config.ts`, `src/main`, `src/preload`, `src/renderer`).
- Typed IPC scaffold (`src/shared/ipc.ts`, preload bridge).
- Pi `createAgentSession` driver in the main process (`src/main/agent/driver.ts`).
- Lifetime-scoped disposables and lifecycle helpers (`src/main/lifecycle.ts`). See [conventions](./conventions.md).
- Chat pane (`src/renderer/chat/Chat.tsx`) with scoped chat CSS, default chat blocks, exact first-party canvas tool block renderers, and a local Iosevka code font asset.
- Sequential extension loader with per-entry lifetimes, error isolation, and manual hot reload (`src/main/extensions/`). See [extension-activation-and-isolation](../decisions/2026-05-30-extension-activation-and-isolation.md) and [manual-reload-extensionsbag](../decisions/2026-05-31-manual-reload-extensionsbag.md).
- **Resource routes substrate** — a single `uix-resource://` custom protocol with route-based dispatch replaces per-resource scheme registration. `src/shared/resource-routes.ts` defines the route model (`path`, `query`, `origin`), normalization, branded `ResourceUrl` encoding, and URL decoding. `src/main/resources/registry.ts` stores `NormalizedResourceRoute` registrations in a Map and dispatches incoming requests by decoding URLs against each route pattern, handing pre-parsed `{ params, query }` to feature handlers via `ResourceRequestContext`. Feature authors declare routes with `createResourceAddressBuilder` from `@uix/api/resources`, which returns `{ route, url(), origin() }` — `route` goes into the contribution, `url()`/`origin()` are consumed by renderer code. The builder is a transitional hybrid; it dissolves into substrate-owned transport when surface contributions land. See [resource-routes](../plans/archive/resource-routes.md).
- Canvas surface: feature-origin `uix-resource://canvas.<workspace>/doc/...` resource route (declared by `createResourceAddressBuilder`), key-addressed mutable latest store, immutable JSON canvas snapshots with anchor-state meta, private turn-state contributions for per-submit and post-agent `uix.turn-state` snapshot refs, snapshot-derived `<canvas-diff>` agent context, anchored canvas tools, whole-document refresh, and an injected writeback shim for form/control and explicit `contenteditable` edits (`src/features/canvas/backend/`, `src/features/canvas/workspace/`, `src/features/canvas/shared/`, `src/main/turn-state/`, `src/main/agent-context/`). Writeback postMessage listener hand-rolls origin/source/key checks — deferred until surface contributions provide substrate-owned message routing. See [canvas-stage-one](../decisions/2026-05-31-canvas-stage-one.md), [canvas-data-channel](../design/canvas-data-channel.md), and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

**In flight:** chat render primitives are being hardened into the shape that later becomes renderer/style/asset contributions; durable transcript identity is the next substrate needed before interactive or stateful chat blocks. Canvas snapshotting now has a substrate-level turn-state registry/coordinator (`src/main/turn-state/`) that owns appending opaque per-feature refs at run boundaries; the canvas registers a `TurnStateContribution` against it and edits its working copy through `CanvasDocumentBuffer` (renamed from `DocumentChannel`/`DocumentBuffer` — a purpose-owned working copy over the store, not a transport or universal document abstraction; "channel" now names only message conduits). Agent context (`src/main/agent-context/`) materializes model-visible `<uix-state>` sections after turn-state appends; the canvas diff materializer reads the current/prior canvas snapshot refs and derives anchored hunks from the version meta instead of consuming pending changes. The next direction is featurifying canvas as a bundled default, not privileged core: canvas contributes a document kind (canonical canvas HTML), surface/writeback behavior, agent anchor tools, source-aware document listeners, and opaque turn state. The turn-state contract still needs the navigation half — preview/restore callbacks beside prepare — which unblocks branch preview/restore and folds in anchor rehydration. The surface↔agent "typed channel" primitive still exists only as the canvas-specific iframe-postMessage writeback; the generic event channel and its in-process transport are not built yet. Filesystem-tool parity and surface versioning/rollback are tracked in [persistence-and-session-foundation](../plans/persistence-and-session-foundation.md), [canvas-data-channel](../design/canvas-data-channel.md), and [pane-and-file-versioning](../design/pane-and-file-versioning.md).

**Deferred from resource routes:** two items captured in [open questions](./open-questions.md) — substrate-owned postMessage origin/source filtering, and the `createResourceAddressBuilder` dissolving when surface contributions land (the builder's `url()` and `origin()` become substrate-internal at that point).

What's _next_ lives in the plans backlog ([`../plans/backlog.md`](../plans/backlog.md)); per-decision history is in [`../decisions/`](../decisions/).
