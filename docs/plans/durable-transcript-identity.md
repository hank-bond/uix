# Spec: durable transcript identity

Rich conversation blocks need stable identity before they can safely own state or actions. A choice/input block, expanded tool row, annotation, scroll target, or cross-pane backlink needs an address that survives replay and branch navigation. The renderer should not learn the difference between provisional live ids and durable session ids; main owns that aliasing and all durable display/interaction state.

This follows the transcript normalization work in [conversation-render-primitives](../design/conversation-render-primitives.md): main already normalizes pi live events and persisted session entries into UIX `TranscriptItem`s. This plan makes those item ids usable as state/action keys.

## Invariant

- Renderer sees one opaque `item.id` and sends actions back with that id.
- In-flight live items may use provisional ids generated by UIX main.
- Main may keep the renderer-facing live `item.id` provisional for row stability; it must canonicalize that id internally before durable state writes.
- Replayed items use canonical pi-session-derived ids directly because no provisional live row exists.
- Main resolves any incoming renderer id through its provisionalâ†’durable alias map before reading or writing display/block state.
- Durable display/block state is backend-owned and keyed by canonical transcript id; renderer state is a cache only.

## D0 â€” Session append observation

Wrap the `SessionManager` append methods UIX needs before handing the manager to pi. For transcript identity, the minimum is `appendMessage`; install the wrapper in the driver after opening/creating the manager and before `createAgentSession` receives it. Add `appendCustomMessageEntry` only once a concrete custom-message consumer needs live durable ids before pi exposes a cleaner post-persist event.

The wrapper calls the original method, receives the durable id, reads the appended entry with `getEntry(id)`, and notifies a UIX-owned observer. This is a local adapter over pi's current API, not a message-content mutation and not a session-file write.

Prefer object identity for normal messages: the same `AgentMessage` object is emitted on `message_end` and passed into `appendMessage(message)` in current pi, so main can keep a `WeakMap<object, provisionalId>` without copying or retaining message content. If pi later emits a post-persist `session_entry_appended` event with correlation metadata, replace this wrapper with the official event.

## D1 â€” Provisional id aliases

When main creates a live transcript item, it generates a provisional id and records enough correlation to canonicalize it later:

- user/assistant/tool-result message rows: `WeakMap<messageObject, provisionalId>` until `appendMessage(message)` returns the durable entry id;
- assistant tool rows: map `toolCallId -> provisionalToolId`; when the assistant message entry is observed, derive the durable tool id as `<assistantEntryId>:tool:<toolCallId>`;
- custom messages: UIX-authored custom blocks should carry an instance id in `details` for block-level correlation; transcript identity still canonicalizes to the custom-message session entry when that entry is observed. Generic third-party custom messages may need the future pi post-persist event because `appendCustomMessageEntry(customType, content, display, details)` does not receive the same `CustomMessage` object that `message_end` emitted.

When a durable id is observed, main records `aliases.set(provisionalId, durableId)` and migrates any pending display/block state from the provisional key to the durable key. The renderer does not receive a rekey event unless we later decide renderer-local durable state exists, which should be avoided.

## D2 â€” Backend-owned display/block state

Add a small main-owned state store for display/block state keyed by canonical transcript id. Start with the first state the rich-block work needs, likely tool expansion or a choice block's submitted/selected state; do not build a generic database ahead of a concrete consumer. The first implementation can be a UIX sidecar under the state root; later it may move into pi `CustomEntry` records or a hosted store behind the same main-owned interface.

Renderer actions flow through main:

```text
renderer action { id, ... }
  -> main resolves aliases.get(id) ?? id
  -> main validates and persists/updates state under the canonical id
  -> main emits an updated transcript/view item or feed event
```

If an action arrives before the durable id exists, main may hold pending state under the provisional id and migrate it when the alias completes, or reject actions that require durability. Interactive blocks that affect the agent should generally require a durable canonical id before conversion into a tool result or user message. The `complete` flag remains about item content/progress, not proof that durable identity has already been observed.

## D3 â€” Replay joins transcript items with state

History replay should emit canonical ids directly from pi session entries and join any stored display/block state before sending items to the renderer. Live items should eventually be updated from the same joined state path once their canonical id is known, but without requiring the renderer to process a rekey.

For tool rows replayed from assistant/tool-result entries, use the same durable id derivation as live canonicalization: `<assistantEntryId>:tool:<toolCallId>`.

## Boundary

This plan does not build render registries, component APIs, or the choice/input block itself. It only establishes the identity and state substrate those blocks need. The renderer remains a consumer of opaque ids and normalized transcript/view items.
