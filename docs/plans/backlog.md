# Backlog

Compact seeds. Keep enough context that a future reader can recover the trigger, constraint, and likely shape without archaeology. Promote a seed to its own plan once it needs sections, ordered units, acceptance criteria, or design rationale that no longer fits comfortably here; delete the seed here when it graduates.

- **Surface contributions + workspace layout** — done: features export `defineSurface` contributions from their workspace dirs; `layout.tsx` composes the flat list and `SurfaceMount` hands each surface its typed channel client.
- **Typed channel events** — done: shared `ChannelContract`s drive backend handlers (`withHandlers`), typed frontend clients (`createChannelClient`), and typed event publishers (`FeatureEventPublisherFactory`); events are schema-validated on both publish and subscribe.
- **Iframe surface transport** — the containment mode for foreign/generated/executable surface code (shelved when the Host/Workspace iframe collapsed — see [workspace-runtime-foundation](./archive/workspace-runtime-foundation.md)); everything in [runtime-surface-composition](./runtime-surface-composition.md) is trusted first-party page-realm code, so this waits for the first untrusted surface. Related: feature-agent link metadata on surfaces, later.
- **`uix-core` embedded-pi config** — orientation block + doc map + smoke-test tools injected into the cockpit's agent.
- **Agent tool contributions from features** — done: manifest features register pi tools into the owned session via `AgentToolRegistry`.
- **Public feature channel contracts** — establish the convention/mechanism for a feature to expose a stable typed API other features can deliberately depend on: e.g. a calendar feature publishes `calendarChannels` (requests + events) as its public contract, and sibling features/surfaces can import that contract without reaching through ad hoc source paths. Mechanics exist in the channel substrate; missing piece is packaging/discovery/import ergonomics and the rule for what counts as a feature's public API.
- **Replaceable cross-feature settings editor** — apply the command-palette pattern to settings: normal `ctx.settings` / `useFeatureSetting` handles stay owner-scoped, while the substrate exposes an intentional cross-feature editor axis consisting of serializable setting descriptors, complete scope values/defaults, structural and referential diagnostics, draft validation, and constrained whole-scope candidate replacement rather than other features' raw handles. An ordinary ambient feature renders the shipped modal and remains removable/replaceable; each feature may still build a richer scoped editor for its own domain. The first referential case is `settings.keybindings`, whose dynamically validated action-id map must retain well-formed inactive ids but include them in an unresolved projection so the human can correct/delete typos or keep dormant bindings. Until this is promoted, humans and agents edit `uix.workspace.json` and reload. Before implementation, settle descriptor presentation metadata, treatment of secret/non-editable settings, malformed external-edit recovery, and whether custom setting controls are links/actions into feature-owned UI rather than generic schema widgets.
- **Sharpen `@uix/api` import boundaries** — fast follow after the settings API naming cleanup. The root `@uix/api` barrel is fuzzy: it behaves like a backend/feature-entry convenience barrel, while renderer-facing APIs live under `@uix/api/workspace` because they import React. Make the semantic import paths explicit: prefer focused submodules (`@uix/api/feature`, `@uix/api/settings`, `@uix/api/channels`, `@uix/api/workspace`) and either shrink/remove the root barrel or define it as a deliberately tiny React-free shared surface. Avoid adding `@uix/api/backend` unless `feature` proves too narrow.
- **Tracked document publication + external edit absorption** — grow the document store/buffer model so a document can be published to a visible workspace path that the agent, user, bash tools, or external apps can edit. UIX watches only tracked published paths, not arbitrary globs; external writes import into the document store as new current versions/snapshots, publish channel updates to surfaces, and record version refs through turn-state for branch rollback. The eventual git-backed document store supplies snapshot compression/diffing/history; the filesystem path is just the local publication surface. Hydration should be able to compare tracked published files against the latest known version and import changed disk content on demand/startup, so correctness does not rely solely on watcher delivery.
- **Branchable application persistence with DoltLite** — offer apps built on UIX an optional feature-facing relational persistence store with commit, rollback, branch, diff, and eventual merge semantics. DoltLite is the leading embedded local implementation candidate because it combines a SQLite-shaped application boundary with Dolt-style version history; verify Node/Electron bindings, cross-platform packaging, transaction/concurrency behavior, merge/conflict APIs, migration compatibility, licensing, maturity, and a hostable implementation path before promoting this seed. Keep the authorities explicit: Git owns workspace source and `uix.workspace.json`, DoltLite owns application data, and Pi session branches own conversation plus feature turn state. A future workspace checkpoint may coordinate immutable refs from all three without claiming a distributed atomic transaction. Monaco and agent file edits should share the Git checkpoint path described in [pane-and-file-versioning](../design/pane-and-file-versioning.md); per-agent worktrees and matching data branches remain future integration capability, not a UIX-core multi-agent orchestrator.
- **Packaged-binary feature templates** — [runtime-surface-composition](./runtime-surface-composition.md) S5 scaffolds new workspaces from the repo's `src/features/`; a packaged app needs the templates shipped as readable source under `resourcesPath` (they double as the worked example for agents authoring features). Lands with the electron-builder/packaging arc, which doesn't exist yet.
- **Session inspector / debug mode** — menu toggle rendering `display: false` custom messages (and other hidden entry streams) in the conversation pane; renderer-only, the transcript items already carry customType/content/details.
- **Transient notices / toast rendering** — provide a substrate channel for backend-originated ephemeral notices and a frontend invocation path that can feed the same notice model. The producer side should stay simple; the unresolved part is renderer ownership and placement. Options still open: one global workspace renderer, feature-selected renderers (for example Chat showing agent notices in a small area), or routing different notice types/scopes to different consumers. Also settle whether frontend notices stay local or submit through a channel handler, how duplicate rendering is prevented when multiple consumers exist, and which notices target one client versus every attached client. Notices are transient presentation, not session entries or workspace settings.
- **Agent-assisted runtime recovery** — turn captured UIX failures into easy handoff points for the vibe-coding audience rather than requiring DevTools or copied stack traces. Feature activation/reload failures, surface error boundaries, channel/backend failures, and recognized inference-provider errors should produce a structured diagnostic with both a stable searchable error `code` and a unique per-occurrence `diagnosticId`. Mint the id at the first capture boundary and preserve it through propagation; each layer may append bounded, timestamped context frames relevant to the failure—owner, phase, operation, feature/session/generation/cell identity, validation paths, and safe state refs—without replacing the original cause or dumping arbitrary application state. The diagnostic carries a human-safe summary, expandable technical details, cause chain, stack, and relevant runtime/log context, all sanitized of credentials and other secrets. The substrate-owned recovery surface offers dismissal, an optional “What were you doing, and what did you expect?” input, and **Ask agent to fix**. The human explanation becomes an ordinary visible message; the complete sanitized diagnostic rides as hidden model-visible context, giving the agent an edit → reload/test → verify loop without cluttering the transcript. Do not auto-run by default. If inference itself is unavailable, the human must first select a working model; recovery can then fork from the failed branch and ask that model to diagnose the original provider/configuration failure. Keep this a diagnostic handoff, not an agent-side UI mutation API: the agent repairs files or the owning source of truth.
- **Cmd+R routes through substrate reload** — Electron's native page reload currently remounts the renderer but does not re-read the manifest, re-activate backend features, refresh workspace settings from disk, or delegate to pi `session.reload()`. Intercept Cmd/Ctrl+R for workspace windows and invoke `window.channels.reload()` / `Channels.reload` instead, then let the existing `surfaces_changed` path refresh the page composition. Keep a separate devtools hard-refresh escape hatch if needed during development.
- **Replace `ReservedFeatureIds` with registration-order collisions** — substrate claims should be enforced by actually registering the claimed thing first (substrate registers before features, registries throw on duplicates), not by a hand-maintained list that can drift. Workspace settings scopes already work this way (namespaces hydrate at startup; `WorkspaceSettingsStore` throws on duplicate scope ids). Migrate `agent` by having the channel registry rely on the substrate's own prior registration colliding naturally. `uix` is the hard case: it's a _prefix_ claim on session entry types (`uix.turn-state`), with no registered object to collide with — needs an explicit prefix-claim mechanism or stays listed. The list shrinks as claims migrate; delete it when only registrable claims remain.
- **Live agent message taps** — add a feature-facing, read-only contribution for reacting to messages/turns only as they occur; the current renderer agent feed and substrate-internal Pi installers are not that backend feature API. Taps never rerun during session restoration. A tap may update feature working state, but anything that must survive reload, session switch, restart, or branch rewind must reach `uix.turn-state`—inline or through stable referenced ids—at an appropriate lifecycle boundary; uncommitted buffer state is intentionally ephemeral. Restoration consumes feature-scoped turn-state history instead of regexing historical messages, which avoids replaying side effects and remains valid after compaction. Settle finalized-versus-persisted event timing, ordering relative to turn-state preparation, and the minimal typed event vocabulary before promotion.
- **User-action event log on CustomEntry** — append-only `pi.appendEntry` streams folded by D3 reducers; model-invisible, branch-aware, no second database. Keep low-frequency (whole session file loads into memory); leaf discipline unresolved for concurrent writers.
- **Fan-out one-off mode** — prepared root per app variant (system prompt + tool contracts + vocabulary + seeded state entries), fork per invocation, never rejoins; sibling branches make every run inspectable and identical prefixes align with provider prompt caching. Needs lifecycle design (fork-per-invocation creation/disposal, concurrency), not new messaging.
- **Docs + examples** — flesh out `src/docs/` stubs and seed `examples/` as primitives land.
