# Surfaces load at runtime through the manifest pipeline

**Surfaces are a contribution facet; identity lives in the module.** `FeatureContributions` gains `surfaces?: readonly string[]` — entry-file references resolved against the feature entry's directory, mirroring how the manifest references feature entries (strings now; an object form can arrive with iframe containment later). The backend registers the resolved paths into a `SurfaceRegistry` facet (per-feature bag, cleared on reload, like every other facet) and states nothing else: the surface's name, contract, and render live only inside the frontend module, whose **default export** is the result of `defineSurface` — mirroring feature entries default-exporting their `FeatureDefinition`. A runtime loader needs a convention for which export is the surface, and a default export is the one an agent cannot misname. The renderer validates after import that the module belongs to the feature the registry says owns it.

**Contracts carry their owner.** `ChannelContract` gains the owning channel id (`feature`), stated once where the contract is defined. `defineSurface` drops its positional `featureId` — the client is minted from the contract's own id — and backend registration checks the contract's id against the registering feature. This is the general convention: every fact is stated in exactly one place and every binding site can check it loudly, because the authors of new features are agents and the substrate should catch their mistakes, not trust their consistency. (Chat binding the substrate-owned `agent` contract is why the surface owner and the channel owner are distinct facts.)

**Delivery: bundled ES modules on the substrate origin.** A substrate-owned resource route (reserved `uix` id) serves each declared entry as an ES module at `uix-resource://uix.<ws>/…?v=<contenthash>`, esbuild-bundled on demand: feature-local imports bundle in (including anything in a workspace-local `node_modules`, which esbuild resolves naturally), the shared-instance set stays external, and `.css` imports stay external with their import attributes and a content-hash query (verified against esbuild 0.21). The route also serves the feature's directory read-only for CSS and assets, so relative URLs and `url()` references resolve natively. Content-hash busting is what makes `/reload` pick up agent edits — a changed source is a new URL, so the browser module cache can never serve stale code.

**The origin is the security boundary; CSP is aimed at it.** The page keeps today's posture — `default-src 'self'`, feature-origin resource hosts (`<featureId>.<ws>`) loadable only in frames — and `script-src`/`style-src` widen to exactly one additional origin: `uix-resource://uix.<ws>`. Only substrate-compiled surface modules can enter the page realm; canvas docs, snippets, and all general content stay frame-only. Surface routes send no CORS headers, so cross-origin content (a canvas iframe) cannot even fetch the modules. Surface code runs directly in the page realm: it is trusted first-party code the manifest author opted in, the same trust the loader already extends to the backend half — shadow DOM was never a security boundary, and real containment remains the iframe surface mode, deferred until foreign/generated surface code exists.

**Shared instances via import map.** `index.html` declares an import map routing the blessed bare specifiers — `react`, `react/jsx-runtime`, `react-dom`, `typebox`, `@uix/api/workspace` — to shim modules on the substrate origin that re-export the page bundle's live instances. One React instance is guaranteed (hooks in feature components just work), and `@uix/api` becomes genuinely value-importable by workspace code. The backend twin is a jiti alias table in the feature loader mapping `@uix/api` (and `typebox`) to the compiled-in implementation, closing the same gap for feature backend code.

**CSS is platform CSS module scripts.** Feature styles are `import sheet from "./x.css" with { type: "css" }` — the web standard, natively executed by Chromium (Electron 42 is far past support thresholds); the bundler only passes the import through. Sheets are handed to `defineSurface({ styles })` and the mount adopts them into `document.adoptedStyleSheets` on mount and removes them on unmount — reload-safe (no leaked link tags) and already shaped for per-shadow-root adoption if shadow surfaces arrive. No CSS machinery is invented: no style-inject helpers, no sibling-file emission, no link-tag conventions.

**There is no builtin concept.** An app-shipped default feature is not a special registry state — it is an ordinary manifest reference. In the repo, the dev workspace manifest points at `src/features/` source directly; a packaged app's templates are just directories its create-new flow copies into the workspace and references from the fresh manifest (reaffirming stamp-at-creation from [workspace-manifest-not-discovery](./2026-07-02-workspace-manifest-not-discovery.md)). At the end of this arc `bundled.ts` and the static `layout.tsx` import list are **deleted**, not kept as fallbacks — a fallback would resurrect a default the user deliberately deleted, the same smell already rejected for discovery.

**Layout v1: a manifest-ordered resizable row.** Surfaces render left-to-right in composition order (manifest order, then declaration order within a feature) in a horizontal panel row with draggable dividers (`react-resizable-panels` — the reference pattern, not a hand-rolled splitter). Ratios persist to `localStorage` keyed by workspace — renderer-owned presentation state per [one-owner-per-state](./2026-06-09-one-owner-per-state.md), not manifest content. Each surface mounts inside an error boundary: a throwing surface renders an error card naming the feature and entry, the frontend twin of the loader's `failed[]`.

**Rejected.**

- _A second protocol scheme for substrate content_: origin is scheme+host and the host partition already exists; a second scheme duplicates registration machinery without adding a guarantee CSP + no-CORS doesn't already give.
- _A per-file transform server (vite-dev-style)_: native module graphs need import-specifier rewriting for cache busting, which is exactly the cleverness bundling avoids — one URL, one hash per surface.
- _Sibling CSS emission + `<link>` injection_: what bundlers do for pages, but it invents registry-carried stylesheet URLs and link lifecycle management that CSS module scripts get from the platform.
- _A builtin/compiled-in flag in the surface registry or loader_: see above; manifest references subsume it entirely.
- _Surface names in the backend facet_ (`{ name, entry }`): the same fact stated twice, an agent-facing consistency trap, with no backend consumer that needs names before the frontend loads.
- _HMR for feature surfaces_: the loop is edit → `/reload`, symmetric with feature backends; a consistent pure-reload dev mode may later replace HMR for the page itself.
